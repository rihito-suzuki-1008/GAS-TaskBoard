作成日: 2026-07-03
ベースライン: commit `7fa4148`(セルフ登録・コメントスレッド/メンション・Slack通知まで実装済み)

## 0. 前提(実装前に必ず読む)

- `AGENTS.md` / `DESIGN.md` / `docs/02仕様書/` を必読。特に:
  - 保存系は既存の保存キュー(`queueNodeMutation` / `queueBackgroundMutation`)経由
  - 親ノードの進捗・ロールアップ期間は保存しない(動的計算)
  - 楽観的競合検知は Nodes のみ / 新規シートには UpdatedAt 照合を追加しない
  - `prompt()`/`confirm()`/`alert()`・ネイティブ`<select>`/`<input type="date">` 禁止、トークン以外の色/px禁止

### 本改修の最重要原則: WBSシートは「一方向の出力アーティファクト」

- アプリは `WBS` シートを**一切読まない**。正本は常にアプリの各シート
- `WBS` は `HEADERS` / `ensureSchema_` / `loadAll` の対象に**含めない**
- WBSシート上の手編集は次回出力で消える(確定仕様)。シート上部に「このシートはアプリから自動生成されます。手編集は次回出力で上書きされます」の注記行を出力する
- AGENTS.md ルール9(日付はテキスト書式)は**アプリのデータシートに対するルール**であり、WBSシートには適用しない(WBS内の日付セルは日付値+表示書式でよい。日数式を成立させるため)

## 1. データモデル拡張

すべて既存列の**末尾に追加**(`ensureSheet_` が不足列を自動追加。既存案件のマイグレーション不要)。

### 1.1 Nodes 追加列

| 列名 | 型 | 内容 |
|---|---|---|
| Deliverable | 文字列(任意) | 成果物。WBSの「成果物」列に出力 |
| Note | 文字列(任意) | WBS出力用の短い備考。※`Description`(アプリ内の詳細説明)とは役割を分ける【決定④】 |
| Progress | 数値(任意) | 手動進捗率。**0/15/30/45/60/75/90/100 のみ**(サーバーで値域検証)。末端ノード専用の入力値 |
| IncludeInWbs | 真偽値(任意) | WBSに反映するか。**空欄は TRUE として解釈**(既存全行が自動的に「反映する」になる) |

### 1.2 Members 追加列

| 列名 | 型 | 内容 |
|---|---|---|
| Company | 文字列(任意) | 会社名。メンバーダイアログに入力欄を追加 |

### 1.3 新規シート(3つ)

いずれも **LockService 直列化のみ・UpdatedAt照合なし**(AGENTS ルール15の非対称性を維持。実装時にルール15の列挙へ追記すること)。

**ActivityLog**(追記専用):

| 列名 | 内容 |
|---|---|
| LogId | UUID |
| NodeId | 対象ノード |
| Field | `status` または `progress` |
| OldValue / NewValue | status→ColumnId、progress→数値 |
| NewValueIsDone | 変更後が「完了状態」か(status→完了列、progress→100)。**記録時点のスナップショット**(後で完了列が変わっても履歴の意味が壊れないように) |
| ChangedAt | ISO日時文字列 |
| ChangedBy | MemberId |

**Milestones**: MilestoneId, Name, Date(YYYY-MM-DD), Note, SortOrder
**Meetings**: MeetingId, Name, Schedule(自由記述。例「毎週木曜 13〜14時」), Note, SortOrder

## 2. 進捗率モデルの変更(最重要・単独でリリース可能な単位)

### 2.1 基本構造

- **末端ノードのみ** `Progress`(手動8段階)を持つ。これは「ユーザー入力値」であり派生値ではない(AGENTS ルール1に抵触しない。実装時にルール1へ但し書きを追記)
- **親ノードの進捗は従来どおり子の平均を動的計算・非保存**(ルール1の核心は不変)
- 末端の実効進捗 = `完了列にいる ? 100 : (Progress が有効値ならその値 : 0)`
  - Progress 列が空の既存データは「完了列=100 / 他=0」となり**現行挙動と完全一致**(マイグレーション不要)
- 変更箇所: サーバー `computeDerived_`(`src/05_Payloads.js`)の leaf 分岐、クライアント `recomputeClientDerived`(`src/ClientDataSync.html`)の leaf 分岐、`clientNode_` に progress 関連フィールド追加

### 2.2 不変条件「完了列 ⇔ Progress=100」をサーバーで強制

`applyNodePatch_`(`src/01_NodeApi.js`)で正規化する。**すべて同一パッチ=1行書き込み**なので競合検知・保存キューへの影響はない。

| 操作 | サーバーの正規化 |
|---|---|
| ステータスを完了列へ変更 | 同時に Progress=100 を書く |
| Progress=100 を設定 | 同時にステータスを完了列へ変更 |
| 完了列から他列へ変更 | **Progress を 90 に戻す**【決定①】(パッチに明示的な progress 指定があればそちらを優先。ただし100は不可) |
| パッチが「Progress=100」と「非完了ステータス」を同時に含む | エラー(「進捗100%は完了ステータスでのみ設定可」) |
| 親ノード(子を持つ)への progress パッチ | エラー(親は自動計算) |

- 注意: 子が全削除されて親→末端に戻った場合、保存済み Progress 値が再び有効になる(仕様として許容。仕様書に明記)
- カスケードリスケジュール・依存関係・競合検知には無関係(進捗は日程計算に使われない)

### 2.3 UI

- 詳細ドロワーに進捗セレクタ(カスタムドロップダウン、8択)を追加
  - 完了列在籍中は「100%」固定で無効化+ヒント「ステータスで管理」
  - 100% を選択した場合はステータス表示も完了列へ即時反映(楽観的更新、サーバーの正規化と一致させる)
  - 親ノードは読み取り専用表示のまま(現行どおり)
- 併せてドロワーに「成果物」「備考」入力欄、「WBSに反映」チェックボックス(既存 `.switch` スタイル、デフォルトON)を追加。すべて `detailDraft` → `saveNodePatch` の既存経路
- 看板カード・リスト・ガントの進捗表示は自動的に中間値対応になる(表示ロジック変更不要のはず。％丸め表示のみ確認)

## 5. WBS出力仕様

### 5.1 出力操作

- トップバー(強制同期の隣)に「WBSに上書き」ボタン → 確認モーダル(「WBSシートを上書きします。シート上の手編集は失われます」。初回はシート作成の旨)→ `serverCall('exportWbs')`
- 実行中はボタンに `submitting` スピナー。完了トースト「WBSシートを更新しました」
- **`pendingMutationCount > 0` または `savingNodeIds` 非空の間は実行をブロック**し「未同期の変更があります。同期完了後に実行してください」を表示(未確定の楽観的状態がWBSに写らないことを保証するのではなく、"シートに未到達の変更が漏れる"事故を防ぐため)
- 二重実行ガード: クライアントのボタン無効化+サーバー側 ScriptProperties のタイムスタンプフラグ(60秒で自動失効)

### 5.2 対象タスクのフィルタ【WBSに反映フラグ】

- `IncludeInWbs = FALSE` のノードは**サブツリーごと除外**(子孫のフラグが TRUE でも出力しない)
- 採番・階層列数(maxDepth)・日付範囲・セクション判定は**フィルタ後のツリー**に対して行う(=「存在しなかった前提」のナンバリング)
- **親の進捗率・ロールアップ期間はアプリの真の値(非表示タスクを含む)をそのまま出力**【決定⑥案a】。「見えている子が全部100%でも親が90%」はあり得るが、それが正しい情報である
- ドロワー側: 祖先に FALSE がいるノードはチェックボックスを無効化し「親タスクがWBS対象外」とヒント表示(自分の値は保持)
- リスト/ガントの行に、WBS対象外タスクの控えめなインジケーター(`--color-text-tertiary` の「WBS外」チップ)を表示

### 5.3 シートレイアウト

行構成(行番号は件数で変動するため builder が動的決定):

1. 注記行(自動生成・手編集消滅の注意書き。薄字)
2. メタ情報ブロック: プロジェクト名(ルートノード名)/開始日・終了日(出力対象タスクの計画min/max)/作成者(エクスポート実行者名)/初回作成日/更新日/バージョン。ラベルセル=濃グレー背景・白字、値セル=淡黄背景
   - 初回作成日・バージョンは `PropertiesService.getDocumentProperties()` の `WBS_CREATED_AT` / `WBS_VERSION`(出力ごとにインクリメント)で管理
3. マイルストーンエリア: 左側に名称一覧、右側日付グリッドの該当列に名称+`▼` マーカー
4. 会議体エリア: 名称+開催ルールのテキスト行
5. ヘッダー2段: 左=列名、右=上段 `7/1` 形式・下段曜日(`木`)
6. タスク行(木順走査。**深さ1ノード=セクション行**として濃グレー/黒背景・白字で左端からガント領域まで全幅横断【決定⑤】。深さ2以下は通常行)

列構成(builder が座標を一元計算し、writer は従うだけ):

```
A         = WBS番号(深さ1が「1」、その子が「1-1」…。ルートは行として出力しない)
B〜       = 階層タスク名列 × maxDepth(フィルタ後の最大深さ。最低3列を保証)
続く12列  = 成果物/備考/会社名/責任者/計画開始日/計画終了日/計画日数/
            実績開始日/実績終了日/実績日数/進捗率/完了フラグ
その右〜  = 日付列(1日1列)
```

- タスク名は該当階層の列にのみ書き、右隣は空セル(長い名前のはみ出し表示はスプレッドシート標準挙動に任せる)
- 固定: 行=ヘッダー2段目まで、列=完了フラグ列まで
- 日付列幅は `setColumnWidths(開始列, 日数, 24)` 一括

### 5.4 各列の内容

| 列 | 出力 |
|---|---|
| 成果物/備考 | Nodes.Deliverable / Nodes.Note |
| 会社名 | 担当者の Company を「・」連結(重複会社は除去、Company 空は除外) |
| 責任者 | 担当者名を「・」連結 |
| 計画開始日/終了日 | Nodes の StartDate/EndDate。**親ノードは自身の日付、なければロールアップ表示期間**。日付値+表示書式 `yyyy/mm/dd` で書く |
| 計画日数 | **式** `=終了日セル-開始日セル+1`(日付が両方ある行のみ) |
| 実績開始日/終了日 | ActivityLog から導出(5.6)。日付値で書く。無ければ空欄 |
| 実績日数 | **式**(実績が両方ある行のみ) |
| 進捗率 | 実効進捗を 0〜1 の数値+パーセント表示書式で出力。**データ検証(プルダウン: 0,0.15,0.3,0.45,0.6,0.75,0.9,1)を設定**。一方向仕様(手編集は次回出力で消える)【決定③】 |
| 完了フラグ | **式** `=IF(進捗セル=1,"完了","")` |

### 5.5 ガントエリアの描画

- **背景はエクスポート時に確定する静的塗り**(値と同サイズの背景色2D配列を組み `setBackgrounds` 1回):
  - 平日=白、土=淡青、日=淡赤(曜日は日付から確定計算)
  - 計画期間(計画開始〜終了)のセル=淡緑で上書き
  - セクション行=濃グレーで全幅横断
- 実績: 実績開始〜終了の各日付セルに文字 `★`(値マトリクスに配置。計画の淡緑と重なっても読める濃色文字)
- マイルストーン: 該当日列に `▼`(マーカー行)+名称
- 条件付き書式は**1本のみ**:「完了フラグ=完了 の行の左側WBS表エリア(A〜完了フラグ列)をグレーアウト」するカスタム式ルール。行ごとにルールを量産しない
- 日付範囲 = フィルタ後タスクの計画min〜max+前後7日。**400列を超える場合は400列で切り詰めて出力し、応答に warning を含めてクライアントがトースト警告**【決定②】

### 5.6 実績日の導出(保存しない。出力時に計算)

`deriveActuals_(logs, ...)`(純粋関数)で NodeId ごとに:

- **実績開始日** = min(最初の status 変更ログ, 最初の「progress > 0」ログ)の ChangedAt(日付部分)
- **実績終了日** = そのノードの**現在の実効進捗が100の場合のみ**、`NewValueIsDone=TRUE` の最後のログの ChangedAt。現在未完了なら空欄(途中の完了→差し戻し履歴は無視)
- ログが1件もないタスクは両方空欄(過去分のバックフィルはしない)

## 6. 実装構造

新規 `src/10_WbsExport.js` に3層で分割:

1. **`exportWbs()`** — エンドポイント。`requireCurrentMember_` → `readAll_()` スナップショット → モデル構築 → 書き込み → `{ ok, version, rowCount, warning? }`
2. **`buildWbsModel_(rows, options)`** — **純粋関数(GAS API に触らない)**。`filterWbsTree_`(IncludeInWbs)、`assignWbsNumbers_`、`deriveActuals_`、レイアウト座標計算、値/背景の2D配列、CF・検証・固定・列幅の定義まで全部ここで確定
3. **`writeWbsSheet_(model)`** — バッチ書き込み専任

- **`tests/wbs_model.test.mjs` を新設**し、採番(フィルタ込み)・可変階層の列座標・実績日導出を Node.js で単体テストする(`tests/reschedule_core.test.mjs` と同じ流儀。GAS依存はモック不要になるよう builder を純粋に保つこと)

### ロック・パフォーマンス方針

- **`exportWbs` は `withLock_` を使わない**。読み取りは `loadAll` 同様ロック無しスナップショットで十分、書き込み先はアプリデータ外。書式込みで数秒かかる処理がスクリプトロックを握ると全ユーザーの保存が「他の操作が進行中です」で弾かれるため(この意図的な例外を AGENTS.md に追記)
- Range 操作は**合計20回程度**に抑える: `clear`+`clearFormats`+既存CF/検証の除去 → `setValues` 1回 → `setBackgrounds` 1回 → フォント/罫線 数回 → CF 1回 → 検証 1回 → 固定 → 列幅(連続グループごと)。**セル単位のループ書き込み禁止**
- シートの削除→再作成はしない(gid が変わり共有リンクが壊れる)。既存 `WBS` シートは中身だけ全消しして再生成

## 10. やってはいけないこと

- アプリが `WBS` シートを読む/`HEADERS`・`ensureSchema_`・`loadAll` に WBS を含める
- 親ノードの進捗・ロールアップ期間・実績日を**シートに保存**すること(すべて動的計算/出力時計算)
