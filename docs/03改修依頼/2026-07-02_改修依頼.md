## 1. セルフ登録(未登録ユーザーのオンボーディング)

### 背景

本アプリは「アクセスしたユーザーとして実行」でデプロイされており、`Session.getActiveUser().getEmail()` によって本人識別が常に成立している(`src/07_Validation.js` の `requireCurrentMember_`)。しかし Members 未登録のユーザーが開いた場合、現状はバナー「メンバー未登録です。保存操作は利用不可」が出るだけの行き止まりで、誰かにメンバー管理画面から登録してもらう必要がある。

全利用者はコンテナスプレッドシートの編集権限を既に持っているため(基本設計書1.2)、セルフ登録を開放しても攻撃面は増えない。

### 仕様

1. `loadAll` の応答が `unregistered: true` の場合、初回描画時に**参加モーダル**(標準モーダル、DESIGN.md 2.2b)を表示する。
   - 表示内容: 「`<メールアドレス>` として参加します」(メールは変更不可)/ 氏名入力(初期値はメールのローカル部)/ 表示色(初回セットアップ画面 `renderSetupColorPicker` と同等のプリセット+詳細オプション)
   - 「参加する」→ 登録 → モーダルを閉じ、そのまま全機能が使える状態にする(`state.currentMember` 確定)
   - 「あとで」→ モーダルを閉じ、従来どおり閲覧専用(既存バナー表示を継続)。再表示導線としてバナー自体をクリック可能にし、参加モーダルを再度開けること
2. サーバーに `joinAsMember(payload)` を新設(`src/02_CollaborationApi.js`)。
   - **Email は必ずサーバー側で `getCurrentEmail_()` から取得する。payload の email は受け取らない**(他人の代理登録をこのAPIでは不可能にする)
   - 受け取るのは `name` / `color` / `clientMemberId`(楽観的更新用のクライアント採番ID。既存の `upsertMember` と同様)
   - `withLock_` 内で実行。Email 重複時はエラーではなく**既存メンバーを currentMember として返して成功扱い**にする(2タブ同時参加の競合で冪等になるように)
   - `getCurrentEmail_()` が空(ドメイン外アカウント等)の場合はエラー
   - 戻り値: `{ ok, members, currentMember }`
3. クライアントは `queueBackgroundMutation('members', 'joinAsMember', ...)` で楽観的に登録する(自分をmembersに即時追加 → 応答で収束)。

### ルール改訂(必須・重要)

`AGENTS.md` ルール6 は「本人識別チェックの例外は初回セットアップのみ」と定めている。本改修で例外は次の2つになる:

1. 初回セットアップ(`setupProject`)
2. セルフ登録(`joinAsMember`。ただし登録できるのはセッションのメール本人のみ)

`AGENTS.md` ルール6 と基本設計書 2.7 をこの内容に改訂すること。**これ以外の例外を追加しない。**

### 受け入れ条件

- [ ] 未登録メールで開くと参加モーダルが出て、1操作で参加でき、直後にタスク編集・コメント投稿ができる
- [ ] payload に他人の email を混ぜても、登録されるのはセッションのメールのみ
- [ ] 2タブ同時に「参加する」を押しても Members に重複行ができない
- [ ] 「あとで」を選ぶと従来の閲覧専用と同じ挙動。バナーから参加モーダルを再表示できる
- [ ] 初回セットアップ(全シート空)のフローは従来どおり変化なし

## 2. コメントのGitHub issue化

発言者の自動記録(AuthorId / AuthorName スナップショット)は実装済みのため対象外。今回は「スレッド」「Markdown表示」「メンション」の3点+表示改善。

### 2.1 データモデル変更(最初に1回で確定させる)

Comments シートに列を**末尾追加**する:

| 列名 | 型 | 用途 |
|---|---|---|
| ParentCommentId | 文字列(任意) | 返信先の CommentId。空なら親コメント |
| Mentions | 文字列(CSV、任意) | メンションされた MemberId のスナップショット |

- ※ コメント編集・削除機能を含める決定になった場合(5章の要決定事項)は、同時に `EditedAt` / `DeletedAt` / `DeletedBy` も追加し、スキーマ変更を1回で済ませること
- `src/00_Config.js` の `HEADERS` / `TEXT_COLUMNS` に追加する。既存シートへは `ensureSheet_` が不足列を自動追加するため移行作業は不要
- 既存行は両列とも空=「親コメント・メンションなし」として自然に解釈される

### 2.2 スレッド(1階層)

- 構造は**親コメント+返信の1階層のみ**。返信への返信は同じ親にぶら下げる(UIの「返信」ボタンは返信コメントにも出すが、`parentCommentId` には親のIDを入れる)。無限ネストは実装しない
- サーバー `addComment`(`src/02_CollaborationApi.js`)に `parentCommentId` を追加。検証:
  - 親コメントが存在し、同一 `NodeId` に属すること
  - 親コメントが返信(ParentCommentId あり)でないこと(1階層の強制)
- 表示(`renderComment` / `src/ClientRenderPanels.html`): 親コメントを Timestamp 昇順、その直下に返信をインデント表示(インデントは `--space-6` 程度+左ボーダー)
- コンポーザー: 各コメントの「返信」ボタン → コンポーザーが返信モードになり「〇〇さんのコメントに返信」表示+キャンセルボタン。返信モード状態は `state.commentReplyTo`(nodeId切替・ドロワークローズでリセット)
- 楽観的更新は既存の `queueBackgroundMutation('comments:{nodeId}', 'addComment', ...)` の payload 拡張のみ。`commentCounts` は返信込みでカウント(現行ロジックのまま)

### 2.3 Markdown表示

- **方針: 全文を既存の `h()` でエスケープした後、ホワイトリストの記法だけを変換する自前ミニレンダラー**。外部markdownライブラリ・生HTMLの許可は禁止(XSS面と Vanilla JS 方針のため)
- 実装: `src/ClientUtils.html` に `renderMarkdown(text)` を追加し、`renderComment` の本文表示で使用。投稿フォームは textarea のまま(表示時のみレンダリング)
- 対応する記法(この範囲のみ):
  - `**太字**` / `*斜体*` / `~~打ち消し~~`
  - `` `インラインコード` `` / ```` ``` ```` フェンスのコードブロック(内部は変換せずエスケープのみ)
  - `- ` / `1. ` の箇条書き・番号リスト(ネスト1段まで)
  - `[テキスト](URL)` と裸URLの自動リンク
  - 改行(単一改行を `<br>` に)
- 非対応(実装しない): 生HTML、画像、見出し、テーブル、引用
- リンクの安全条件: **href は `http:`/`https:` のみ許可**(`javascript:` 等は拒否しプレーン表示)。`target="_blank" rel="noopener noreferrer"` を付与
- コードブロック等のスタイルは `--font-mono` / `--color-canvas` などトークンのみで実装

### 2.4 メンション

- コンポーザーで `@` 入力時にメンバー名のオートコンプリートを表示(既存ドロップダウンの見た目に合わせたフローティングパネル。カーソル位置基準)。選択で本文に `@氏名 ` を挿入
- **保存形式**: 本文はプレーンテキストのまま。加えて payload の `mentions: MemberId[]` を `Mentions` 列に CSV でスナップショット保存(サーバーで Members 照合し、存在しないIDは除外)。氏名変更・メンバー削除後も宛先が壊れないようにするため
- 表示: `Mentions` に含まれるメンバーの `@氏名` をハイライト。自分宛(`currentMember.id` を含む)のコメントは `--color-accent-subtle` の背景で強調
- 将来の Slack 通知(3章)でメンション通知に拡張することを想定し、構造化(Mentions列)は省略しないこと

### 2.5 表示改善

- コメントヘッダに発言者アバター(メンバー色の丸+イニシャル、既存 `renderAvatar` 流用)を追加。Members から消えた投稿者は AuthorName のみ+グレーのアバターで表示

### 受け入れ条件

- [ ] 返信が親コメントの下にインデント表示され、返信への返信も同じスレッドに入る(2階層目が生まれない)
- [ ] 既存コメント(旧形式の行)が親コメントとして正常表示される
- [ ] `<script>` や `javascript:` リンクを含む投稿がテキストとして無害化表示される
- [ ] 太字・コードブロック・リンク・リストが正しく描画される
- [ ] `@` でメンバー補完が出て、投稿後にハイライト表示。メンバー名を変更しても既存コメントの宛先ハイライトが壊れない
- [ ] 楽観的更新: 投稿直後に即表示、失敗時はロールバック+エラートースト(既存挙動の維持)

## 3. Slack通知(ステータス変更)

### 方針

トリガーによるシート監視は行わない(onEdit/onChange はスクリプト自身の書き込みでは発火しないため不成立。ポーリングは遅延と複雑さに見合わない)。**`saveNode` が既に持つ `statusChanged` 判定にフックする**(`src/01_NodeApi.js`)。

### 仕様

1. `src/09_Notifications.js` を新設:
   - `buildStatusChangeNotification_(node, beforeColumn, afterColumn, actor, rows)` — タスク名 / 親パス(祖先の名前を「 > 」連結) / 旧列名→新列名 / 実行者名 / WebアプリURL を含む Slack payload(Block Kit または plain text)を生成
   - `postToSlack_(payload)` — `UrlFetchApp.fetch(webhookUrl, ...)`。`muteHttpExceptions: true`、**try/catch で握りつぶし、通知失敗が保存を失敗させないこと**
2. Webhook URL は `PropertiesService.getScriptProperties()` の `SLACK_WEBHOOK_URL`。**未設定なら通知処理全体をスキップ**(=未設定でも現行と完全に同じ動作。安全にデプロイ可能)
3. **送信はロックの外で行う**。`withLock_` 内では通知データの組み立てまでとし、`saveNode` は `withLock_` の戻り値に通知データを添えて、ロック解放後に送信してからクライアントへ返す。ロック中に UrlFetch(数百ms〜数秒)を行うと他ユーザーの保存を全てブロックするため厳禁
4. 通知対象は v1 では「ステータス変更」のみ。カスケードリスケジュール・タスク追加・メンション通知は対象外(拡張しやすい関数分割にはしておく)
5. README に設定手順(Incoming Webhook の発行、Script Properties への設定、初回の再認可)を追記

### 注意事項

- `UrlFetchApp` 初使用で OAuth スコープが増え、**全ユーザーに1回だけ再認可プロンプトが出る**。リリースノート等で事前周知すること
- Script Properties はシート編集権限を持つ全員が閲覧できる。Incoming Webhook URL はその前提で扱う(投稿しかできないURLなので許容)

### 受け入れ条件

- [ ] 看板のカードD&D / ドロワーでのステータス変更の両方で通知が飛ぶ
- [ ] ステータス以外の変更(名称・日付等)では飛ばない
- [ ] `SLACK_WEBHOOK_URL` 未設定でも一切の挙動変化・エラーがない
- [ ] Webhook URL が無効でも保存は成功し、クライアントにエラーが出ない
- [ ] 通知送信中に他ユーザーの保存がブロックされない(ロック外送信の確認)

## 7. やってはいけないこと

- 保存キュー(`queueNodeMutation` / `queueBackgroundMutation`)を迂回する保存経路の新設
- Members / StatusColumns / Dependencies / Comments への UpdatedAt 照合(楽観的競合検知)の追加 — 競合検知は Nodes のみ(AGENTS.md ルール15)

- `joinAsMember` 以外への本人識別チェック例外の追加
- `prompt()` / `confirm()` / `alert()`、ネイティブ `<select>` / `<input type="date">` の使用
- 日付・日時の Date 型保存(文字列 `YYYY-MM-DD` / ISO 文字列を維持)
- デプロイ設定(実行モード・アクセス範囲)の変更
