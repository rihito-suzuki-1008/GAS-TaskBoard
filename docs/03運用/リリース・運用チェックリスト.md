# リリース・運用チェックリスト

このアプリは社内向けの案件別タスク管理アプリです。1つのスプレッドシートを1案件の正本データストアとして扱い、`WBS` シートは出力専用の成果物として扱います。

## リリース前チェック

### 自動チェック

Pull Requestとmainへのpushでは `.github/workflows/quality.yml` が以下を自動実行します。リリース担当者もデプロイ前にローカルで同じチェックを実行します。

```sh
for f in src/*.js; do node --check "$f" || exit 1; done
for f in src/Client*.html; do node --check --input-type=commonjs < "$f" || exit 1; done
node --test tests/*.test.mjs
git diff --check
```

### 手動スモーク

- 初回セットアップ画面から案件を作れること。
- Members作成後やStatusColumns作成後に意図的に初期化を失敗させても、再試行で重複なく完了できること。
- 旧Nodesヘッダーへ `DraftOwner` / `DraftExpiresAt` / `ActualStartDate` / `ActualEndDate` が末尾追加され、列順を入れ替えたシートでは安全に停止すること。
- メンバー追加、会社名、色変更が保存されること。
- 看板でカードを別ステータスへドラッグできること。
- リストで名称、ステータス、担当、優先度、期間、末端進捗を直接編集できること。
- リスト/ガント/看板でタスク順と親子関係をD&Dで変更できること。
- 子タスクの一部が完了した親が進行中になり、子がすべて完了した親が完了になること。
- 完了列・進行中列を切り替えた際、既存末端タスクの進捗と親ステータスが新しい役割へ即時に収束すること。
- 依存関係追加時と先行タスクの日程変更時に、後続タスクが必要分だけ後ろへずれること。
- 詳細ドロワーの保存ボタン押下直後にドロワーが閉じ、保存失敗時はエラーが出ること。
- 詳細ドロワーを閉じた保存が競合した場合、入力内容を保持したドロワーが再表示されること。
- 詳細ドロワーがデスクトップで説明・コメントとプロパティの2列、960px以下で1列になること。
- 実績日を手動設定したWBSではその期間に `★` が付き、「自動」へ戻すとActivityLog由来の日付へ戻ること。
- 削除が競合した場合、楽観的に消した子孫がその場で復元されること。
- タスク削除後、トーストの「元に戻す」で削除したタスクと子タスクを復元できること。
- 先に削除済みだった子タスクを含む親タスクを削除してUndoしても、先に削除済みだった子タスクが復元されないこと。
- 削除確認後に子タスクが増えた場合、削除を中止して再確認を求めること。
- 名称未入力の作成途中タスクを再読み込み後に復元でき、別ユーザーからは見えないこと。
- Slackアプリ方式／Workflow Builder方式の切り替え、通知種別ごとのON/OFF、用途別接続テスト、失敗状態表示、方式別連携解除が動作すること。
- Slackアプリ方式の送信文面編集・デフォルト復元と、Workflow Builder方式の用途別URL・フラット変数送信が動作すること。
- URL保存後に入力欄が空になっても「設定済み」と固定マスクが残り、空欄で再保存しても既存URLが消えないこと。
- 設定保存と各接続テストで対象ボタンと近接状態欄にスピナーが表示されること。
- Slack設定保存直後に接続テストを押しても、新しく保存したWebhookへ送信されること。
- メンション通知ON時は、Slackアプリ方式では1件に集約され、Workflow Builder方式ではSlackUserId登録済み対象者ごとに1件通知されること。OFF時とメンションなしでは通知されないこと。
- Slack送信に失敗してもタスク保存・コメント保存が成功し、最新の失敗状態が設定画面へ反映されること。
- マイルストーン/会議体がガント上に表示され、会議体の `▼` ホバーで内容を確認できること。
- 100件前後のタスクで、保存、D&D、ビュー切り替え、WBS出力の体感が許容範囲であること。
- 500ノード・コメント累計10,000件相当の検証データで、初期表示がコメント本文全件に比例して悪化せず、詳細のコメントが50スレッド単位でページ表示されること。
- `=1+1`、`+123`、`-123`、`@name` で始まる名称・説明・コメントが数式にならず、そのまま表示されること。
- 2ユーザーの一方が対象シートを並べ替えた直後に他方が保存した場合、`ROW_IDENTITY_MISMATCH` で誤更新せず停止すること。
- 2000-01-01より前、2100-12-31より後、または3,660日超の日程が拒否され、既存の異常値があってもガント幅が3,660日を超えないこと。

## WBS出力スモーク

WBS出力後、実シートで以下を確認します。

- 出力成功後にステージング/バックアップシートが残らず、`WBS` だけが最新内容へ切り替わること。
- 出力途中に例外を発生させても、直前の `WBS` が残ること。
- 2ユーザーが同時に出力した場合、片方が `WBS_BUSY` で安全に中止されること。
- A列のWBS番号が `1-1` のような値でも日付化していないこと。
- 進捗率プルダウンが明細行だけに出て、セクション行やヘッダー行には出ていないこと。
- 進捗率の色が 0/15/30/45/60/75/90/100 の選択肢に対応していること。
- 計画期間の緑より、土日の赤/青が見えること。
- マイルストーンが近接日でも縦にずれて表示されること。
- 会議体は日付グリッドに会議名を書かず、開催日の `▼` だけが出ること。
- 実績開始日/実績終了日がActivityLogから想定通りに出ること。
- 名称未確定のドラフトタスクがWBS番号・期間・進捗へ混入しないこと。
- 400日を超える範囲では警告が表示され、日付列が切り詰められること。

## 権限と配置

- Webアプリは「アクセスしたユーザーとして実行」で運用します。
- 同一Google Workspaceドメイン内のユーザー利用を前提にします。
- 利用者にはコンテナスプレッドシートへの編集権限が必要です。
- 直接シート編集は技術的には可能ですが、アプリ側のバリデーションを迂回します。通常利用者にはWebアプリから操作するよう案内します。
- 案件を増やす場合は、テンプレートのスプレッドシートをコピーし、案件ごとにGASプロジェクト/デプロイを運用します。
- Membersから人を削除してもDriveの共有権限は消えません。案件離脱・退職時は、スプレッドシートの直接共有と共有Google Groupの両方から対象者を外し、必要に応じてSlack Webhookをローテーションします。
- スプレッドシート編集者はコンテナバインドされたコードを変更でき、Script Propertiesを読み出すコードも実行できます。この信頼モデルに置けない利用者へは編集権限を付与しません。

## デプロイ手順

本番デプロイは「ソース確定 → CI成功 → GAS反映 → 既存deploymentIdの更新 → スモーク」の順で行います。`clasp push` と本番デプロイの間に別の変更を混ぜません。

```sh
git status --short
git push origin main
```

mainのQuality checks成功を確認後、同じコミットの作業ツリーで実行します。

```sh
git status --short
clasp push -f
clasp deploy --deploymentId <production-deployment-id> -d "<release note>"
```

スモーク完了後、GASのデプロイ画面に表示されるバージョン、GitコミットID、実施者、実施時刻、結果を運用記録へ残します。失敗時は追加修正を同じデプロイへ継ぎ足さず、直前の安定コミットへ戻して再度CIからやり直します。

デプロイ後は、開きっぱなしの旧クライアントから保存する前に、まずWebアプリを1回リロードして `loadAll` を成功させます。これにより `DraftOwner` / `DraftExpiresAt` / `ActualStartDate` / `ActualEndDate` / `IsInProgressColumn` の既知末尾列追加を完了させます。その後、トップバー表示、ビュー切り替え、保存1件、WBS出力1回を確認します。

## 復旧

- 誤削除はソフトデリートなので、通常画面からは消えますがNodesシート上には残ります。
- 削除直後はトーストの「元に戻す」で復元できます。
- 削除直後以外の復旧が必要な場合は、スプレッドシートの変更履歴、または管理者によるシートメンテナンスで対応します。
- `WBS` シートは出力専用です。シート上の手編集は次回出力で消えます。
- デプロイ後に重大な問題が出た場合は、直前の安定コミットへ戻して `clasp push -f` と本番deploymentIdへの再デプロイを行います。
- `STATUS_DONE_INVARIANT` はStatusColumnsの `IsDoneColumn=TRUE` を厳密に1行へ、`STATUS_PROGRESS_INVARIANT` は `IsInProgressColumn=TRUE` を0〜1行へ直接修正してから再読み込みします。アプリは破損を黙って自動補正しません。
- `SCHEMA_MISMATCH` は列を自動上書きせず停止します。変更履歴またはバックアップで期待ヘッダー順へ戻し、既知の末尾追加だけはWebアプリの `loadAll` に任せます。
- `ROW_IDENTITY_MISMATCH` は、読込後に行の並べ替え・挿入・削除が発生した可能性を示します。再実行を連打せず、画面を同期して対象内容を確認してから保存し直します。
- タイムアウトは失敗確定ではなく「結果不明」です。保存やコメント投稿をすぐ再実行せず、同期またはコメント再読込で反映済みかを先に確認します。

## ActivityLogの退避

ActivityLogはWBSと詳細ドロワーの自動実績開始日・実績終了日の算出根拠であり、アプリは自動削除しません。退避は案件終了後、WBSを確定し、その案件で今後WBSを再生成しないことを案件責任者が確認した場合だけ実施します。

1. スプレッドシート全体をコピーし、ファイル名に案件名と退避日を入れて閲覧制限付きの保管先へ移します。
2. コピー先のActivityLogについて、元シートとデータ行数、先頭・末尾のLogIdが一致することを確認します。
3. 現行ファイルのWBSを最終出力し、PDFまたは変更されない別ファイルとして保管します。
4. 現行ファイルのActivityLogを整理する場合は、ヘッダー行を残してデータ行だけを管理者が削除します。アプリ利用中の案件では実施しません。
5. 実施日、担当者、退避先、削除前行数を案件の運用記録へ残します。

シートのセル数が上限へ近づいているだけで案件が継続中の場合は、ActivityLog単独の削除ではなく、案件スプレッドシート自体の年度分割または新案件への切替を検討します。

## 既知の運用限界

- リアルタイム同期ではありません。保存タイミングで競合検知し、画面へ戻った時または同期ピルをクリックした時に、未保存操作がなければ最新状態を適用します。
- Nodes以外のシートはUpdatedAt照合を行わず、LockServiceによる直列化のみです。
- ActivityLogは追記専用です。WBS出力時と、詳細ドロワーで開いたノードの自動実績日を表示する時だけ読み込みます。案件終了後は上記手順で監査用コピーと最終WBSを確保してから退避します。
- 主な想定規模は100タスク前後です。大幅に超える場合は、描画、WBS書式出力、google.script.runの体感遅延を再評価します。
- コメント本文は初期読込から除外し、件数はNodeId列だけ、詳細は選択ページだけを読みます。ただしコメント索引の走査自体は件数に比例するため、数万〜十万件規模では案件分割を検討します。
- ガント表示は安全のため2000-01-01〜2100-12-31、かつ最大3,660日へ制限します。これはガントのDOM肥大化を防ぐ上限であり、超長期案件は案件またはフェーズを分割します。
- `Nodes` などのデータシートが1000行を超えて自然増加した場合は、追加行のテキスト書式が維持されているか確認します。
