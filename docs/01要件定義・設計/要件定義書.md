**タスク管理アプリ 要件定義書**

Google Apps Script（GAS）版 / コンテナバインド構成

バージョン: 5.0

作成日: 2026-07-01

ステータス: 社内レビュー3回反映済み

# 1\. はじめに

## 1.1 目的

プロジェクト推進にあたり、担当者間でタスクの状況・割り当て・スケジュールを一元管理できる社内向けタスク管理アプリケーションを、Google Apps Script（GAS）およびGoogle スプレッドシートを基盤として構築する。本書は実装担当者に引き継ぐための要件定義書である。

## 1.2 v1からの主な変更点（社内レビュー反映）

初版（v1.0）に対し社内レビューを実施し、「複数人での同時編集」「本人特定」「階層構造とステータスの整合性」「ガントチャートの機能範囲」について前提の甘さが指摘された。本v2.0では以下を見直している。

- 1スプレッドシート＝1案件に固定し、プロジェクト切り替えUIを廃止（旧版では複数プロジェクトを想定していたが撤回）
- リアルタイム同期は行わず、保存時の競合検知のみで対応する方針を明確化
- 利用者はGoogle Workspaceアカウント限定とし、コメント投稿者・更新者を自動識別する設計に変更
- 子を持つノード自身のステータスは看板上の位置付けにのみ使用し、進捗率には影響しないことを明記
- ガントチャートをフル機能化（バーのドラッグ編集、依存関係の連鎖自動リスケジュール、日/週/月ズーム）
- メンバー管理機能を新設（従来は登録手段が未定義だった）

さらに2回目のレビューを経て、以下をv3.0で修正している。ガントの機能範囲（ドラッグ編集・依存関係の自動連鎖リスケジュール等）は削らず、その正しさを担保する方向で対応した。

- 案件コピー直後（メンバー0件・ステータス列0件の状態）でもアプリが起動できるよう、初期セットアップの内容を拡充
- 依存関係の自動連鎖リスケジュールについて、複数の先行タスクを持つ合流構造でも正しく動作するようアルゴリズムを修正
- 自動リスケジュールの挙動を「常に隣接させる」から「整合が崩れたときだけ、崩れた分だけ繰り下げる」方式に統一し、ユーザーが意図的に空けた余白を維持するようにした
- コメント投稿者が退職・削除された場合でも過去のコメントの投稿者名が失われないよう、投稿時点の氏名を記録する方式に変更

さらに3回目のレビューを経て、以下をv4.0で修正している。デプロイ実行モードは「アクセスしたユーザーとして実行」を維持し、ノード削除は物理削除からソフト削除（論理削除）に変更した。

- 新規ノードを作成するAPI（addNode）が定義されていなかったため追加
- メンバーが0件になるとアプリ全体が機能停止するため、最後の1人のメンバー削除を禁止し、自己削除には強い確認を設ける
- 依存関係は末端ノード（タスク）同士のみに設定可能とし、グループ・プロジェクトへの依存設定は不可とした（ロールアップで動的に変わる期間を依存の基準にすると不安定なため）
- 開始日・終了日は必須項目ではないことを明記し、未設定ノードの扱い（ロールアップ計算からの除外、依存関係への参加不可）を定義
- 依存関係を新規追加した時点でも日程の整合チェックとリスケジュールを行うよう変更（従来は日程変更時のみだった）
- ノードの削除を物理削除からソフト削除に変更し、誤削除からの復旧を容易にした

## 1.3 用語定義

| **用語** | **定義** |
| --- | --- |
| ノード（Node） | プロジェクト・グループ・タスクを問わず、階層構造を構成する最小単位。全ノードは同一のデータ属性を持つ。 |
| プロジェクト（ルートノード） | 階層構造の最上位ノード。1つのスプレッドシートにつき1件のみ存在する。 |
| グループ | 子ノードを持つノードのUI上の呼称。データ構造上はタスクと同一。 |
| タスク（リーフノード） | 子ノードを持たないノードのUI上の呼称。 |
| ロールアップ | 子ノードの値（進捗率・期間など）を親ノードに集計して算出する処理。値は保存せず、参照のたびに計算する。 |
| カスケードリスケジュール | 先行タスクの日程変更を受けて、依存する後続タスク群の日程を連鎖的に自動調整する処理。 |

# 2\. システム概要

## 2.1 提供形態

- 1つのGoogleスプレッドシートが1つの案件（プロジェクト）に対応するコンテナバインドのGASプロジェクトとして構築する。案件が増えた場合はスプレッドシートごとコピーし、別アプリとして運用する。
- エンドユーザー向けのUIは、doGet(e) で配信する独立したWebアプリ（専用URL・フルスクリーン表示）として提供する。
- データの実体はスプレッドシートの各シートに保存し、GASがサーバーサイドの役割を担う。

## 2.2 想定利用者・アクセス前提

| **項目** | **内容** |
| --- | --- |
| アカウント | 利用者は全員、会社のGoogle Workspace（組織ドメイン）アカウントでアクセスする前提とする。個人Gmail等の混在は想定しない。 |
| デプロイ設定 | Webアプリは「アクセスしたユーザーとして実行」かつ「同一ドメイン内の全員がアクセス可」で公開する（本人識別を成立させるため必須の設定。詳細は設計書）。 |
| 同時利用人数 | 2〜5人程度。基本的には1人ずつ操作し、まれに複数人の操作が重なる想定。 |
| データ規模 | 1案件あたり、ノード総数は〜100件程度を目安とする。 |
| アクセス制御 | 利用者は全員が閲覧・編集可能（機能・データ単位の権限制限は設けない）。 |

# 3\. 機能要件

## 3.1 データ構造：階層型ノードモデル

案件（プロジェクト）を頂点とした、無制限階層のツリー構造でノードを管理する。ルートノード（プロジェクト）はスプレッドシートごとに1件のみ存在し、初回起動時のセットアップで作成する。

初回セットアップは途中失敗から再開可能とし、既に作成済みの自分のメンバー・初期ステータス列を重複作成せず、不足分を補完して最後にルートノードを作成する。

グループとタスクはUI表示上の呼び分けであり、データモデル上は完全に同一の属性を持つ。子を持つノード自身にも、独立した作業対象としてステータス・担当者・期限を設定できる。ただし3.6の通り、子を持つノード自身のステータスは進捗率の算出には使用しない。

提案書作成（プロジェクト・ルート）

└ 顧客ヒアリング（グループ）

　└ ヒアリングシート作成（タスク）

　　└ 質問項目精査（タスク）

## 3.2 ノード共通属性

| **属性** | **内容** |
| --- | --- |
| 名称（Name） | ノードの名前。必須項目。 |
| ステータス（Status） | 看板ビューの列に対応。子を持つノードの場合、これは看板上の位置付けのみに使われ、進捗率には影響しない（3.6参照）。 |
| 担当者（Assignees） | 事前登録したメンバー一覧（3.3）から複数選択可。主担当者の概念は設けない。 |
| 優先度（Priority） | 「高・中・低」の3段階。 |
| 開始日・終了日 | 任意項目。ガントチャートの表示・依存関係の計算に使用する。未設定のノードはロールアップ（子孫の期間集計）の対象から除外され、依存関係の先行・後続にも設定できない（3.7参照）。子ノードの日程が親の設定範囲を超えた場合、表示上は自動的に拡張される（実際の値の書き換えは行わない）。 |
| 実績着手日・実績終了日 | WBSの実績期間を後から補正する任意項目。両方を設定するか両方を未設定にする。未設定時は従来どおりステータス・進捗変更履歴から自動算出し、予定日程・ガント・依存関係には影響させない。 |
| 進捗率（Progress） | 末端ノードのみ、0/15/30/45/60/75/90/100 の8段階で手動入力できる。親ノードは子ノードの平均を自動算出し、保存しない。 |
| 成果物（Deliverable） | WBS出力に掲載する成果物名。 |
| WBS備考（Note） | WBS出力用の短い備考。アプリ内の詳細説明（Description）とは分けて扱う。 |
| WBS反映（IncludeInWbs） | WBSシートへ出力するかどうか。未設定は反映するものとして扱う。FALSE のノードはサブツリーごとWBSから除外する。 |
| 説明（Description） | ノードの目的や詳細を記載する自由記述欄。 |
| コメント（スレッドログ） | メモや進捗報告を親コメント＋返信1階層で記録。投稿者はログインアカウントから自動識別され、手動選択は不要（3.5）。本文は安全な範囲のMarkdown表示とメンションに対応する。 |

## 3.3 メンバー管理

担当者・コメント投稿者の候補となるメンバーを、専用の管理画面で登録・編集・削除できる。

- 登録項目：氏名、会社のメールアドレス（Google Workspaceアカウントと一致させる）、表示色。
- メンバーを削除した場合、全ノードの担当者欄から当該メンバーを自動的に取り除く。過去のコメントの投稿者表示（氏名）は、投稿時点のスナップショットとして残るため、削除後も消えない。
- メンバーの追加・編集・削除は、他の操作と同様に全利用者が行える。管理者だけに制限する権限区分は設けない（4.5のアクセス方針と一貫）。
- 案件をコピーして新規に使い始めた直後（メンバー0件の状態）は、初回セットアップ時にアクセスした本人が自動的に最初のメンバーとして登録される（4.4参照）。
- ログインユーザーのメールアドレスがメンバー一覧に未登録の場合は、参加モーダルから自分自身をメンバー登録できる。登録されるメールアドレスはログイン中のセッションメールに固定し、ユーザー入力では変更できない。
- メンバーが0件になると本人識別ができなくなり、アプリ全体が保存不能に陥るため、最後の1人のメンバーを削除する操作は禁止する。自分自身のメンバー情報を削除する操作には、通常より強い確認（意図の再確認）を挟む。
- ログインユーザーのメールアドレスがメンバー一覧に未登録で、参加モーダルを「あとで」閉じた場合は、閲覧専用として利用できる。バナーから参加モーダルを再表示できる。

## 3.4 3種のビュー

### 3.4.1 看板ビュー

- ステータスを列として、ノードをカード表示する。
- 列（ステータス）はユーザーが自由に追加・名称編集・並べ替え・削除できる。
- 列のいずれか1つを必ず「完了列」として指定する（0個・複数指定は不可）。
- ノードが1件でも所属している列は削除できない（先に別の列へ移動させる必要がある）。
- デフォルトでは末端ノード（子を持たないノード）のみをカードとして表示し、切り替えで子を持つノードも表示できる。
- カードのドラッグ&ドロップでステータスを変更できる。

### 3.4.2 ガントチャート（タイムライン）ビュー

- 階層構造をツリー表示し、各階層を折りたたみ／展開できる（横スクロール時も左のツリー部分は固定表示）。
- タスク行の上に、マイルストーンと会議体を専用レーンで表示する。近接するマイルストーンは表示粒度と縮尺に応じて縦位置をずらし、会議体は開催日にコンパクトな `▼` を表示する。
- バーをドラッグして日程を移動、またはバー端をドラッグして開始日・終了日を個別に変更できる。
- 親ノードのバーは手動設定した期間を基本とし、子ノードの期間がそれを超える場合は表示上自動的に拡張される。親の期間が子より広い場合は「子に合わせる」ボタンで縮小できる（拡張・縮小いずれも親の実データの書き換えは、ユーザーの明示操作を伴う場合のみ発生する）。
- タスク間の先行・後続関係を矢印で接続して表示する。
- 先行タスクの日程変更（延伸・繰り下げ等）により、後続タスクの開始日が先行タスクの終了日より前になってしまう（整合が崩れる）場合、その崩れた分だけ後続タスクの日程を繰り下げる。ユーザーが意図的に空けていた余白（バッファ）がある場合はそのまま維持し、常に隙間なく隣接させるような処理は行わない。
- この繰り下げは、依存関係で連鎖するすべての後続タスクに自動的に伝播する（例：A→B→CでAが延びれば、BもCも連動して後ろにずれる）。
- 後続タスクが複数の先行タスクを持つ場合（合流構造）、開始日はすべての先行タスクの終了日のうち最も遅いものを基準に整合を判定する。
- 表示粒度は日・週・月でズーム切り替えできる。
- 日表示の背景色は、土曜日を水色、日曜日および祝日をピンク、今日をオレンジ系とする。今日の色を最優先し、土日・祝日と重なっても混色しない。
- 日付ヘッダーから1日または期間を選び、案件共通の「稼働日」「休日」へ手動変更できる。通常は土日と日本の祝日（振替休日・国民の休日を含む）から自動判定し、「標準に戻す」で手動設定を削除する。手動休日・稼働日には「夏季休暇」「休日出勤」等の任意名称を付けられる。
- 案件カレンダーはガントの表示上の区別だけに使用し、タスク期間・日数・依存関係の自動リスケジュール・WBS出力には影響させない。個人の休暇予定は本機能の対象外とする。
- 各ノードの進捗率をバー上に表示する。
- 期限超過・期限間近のノードを色やアイコンで強調表示する。

_※ ガントチャートは本アプリの中で最も実装難易度が高い機能である。詳細な実装順序・検証観点は設計書3.3に記載する。_

### 3.4.3 リストビュー

- 階層構造をインデント付きのツリー形式で一覧表示する。
- 列（ステータス・担当者・優先度・期間・進捗率など）を表形式で表示する。
- 名称、ステータス、担当者、優先度、期間、進捗率は表の中で直接編集できる。通常時はチップや日付表示として見せ、クリック後だけ入力UIを開く。
- 行のドラッグ&ドロップで、表示順や親子関係を変更できる。
- 期限超過・期限間近のノードを強調表示する。

### 3.4.4 ビュー共通機能

- キーワード、階層ノード、ステータス（複数選択可）、担当者、優先度、期間による絞り込み（フィルタ）を各ビュー共通で提供する。
- リスト・ガント・看板の各ビューで、ドラッグ&ドロップにより表示順や親子関係を変更できる。

## 3.5 タスク詳細ドロワー

いずれのビューからもノードをクリックすると右側の広幅詳細ドロワーが開く。デスクトップでは左側を名称・説明・コメントの主領域、右側をステータス等の属性領域とする。狭い画面では1列へ切り替える。

- 名称、ステータス、担当者、優先度、開始日・終了日、実績着手日・実績終了日、成果物、WBS備考、WBS反映、説明
- 進捗率（末端ノードは8段階で編集可、親ノードは自動算出値の閲覧のみ）
- 親ノード・子ノードへのナビゲーション
- コメント欄：投稿者・日時は自動記録されるスレッドログ形式。親コメントへの返信は1階層のみとし、返信への返信も同じ親コメント配下に表示する。本文表示は太字、斜体、打ち消し、インラインコード、コードブロック、リスト、http/httpsリンク、単一改行に対応する。`@` 入力でメンバーを補完し、投稿後はメンションを強調表示する。
- 実績日を手動設定した場合はWBSの実績日と `★` の期間に優先して使う。「自動」へ戻すとActivityLogからの従来算出へ戻る。

## 3.6 進捗率の算出

- 末端ノード（子を持たないノード）：ステータスが「完了列」であれば100%、それ以外は手動入力された8段階進捗を表示する。未入力の場合は0%とする。
- ステータスを完了列へ変更した場合は進捗を100%にし、進捗100%を選んだ場合はステータスも完了列へ移動する。完了列から他列へ戻した場合は進捗を90%へ戻す。
- 子を持つノード：直下の子ノードの進捗率の単純平均を再帰的に算出する。子を持つノード自身のステータスは進捗率の計算に一切使用しない（看板上の位置付けとしてのみ機能する）。
- 親ノードの進捗率は保存しない。子が全削除されて再び末端になった場合、保存済みの手動進捗値が再び有効になる。
- 親ノードのステータスは直属の子から補正する。すべての子が完了なら親も完了、進行中の子または完了済みの子が1つでもあり全子完了ではない場合は親を進行中にする。
- 完了列または進行中列の役割を別列へ切り替えた場合、同じロック内で末端ノードのProgressを新しい完了列に合わせて100/90へ正規化し、親ノードのStatusColumnIdをボトムアップで再計算する。応答には役割変更の影響を受ける全ノードを含める。

## 3.7 依存関係のルール

- 自己参照（自分自身への依存）は禁止する。
- 同一組み合わせの重複依存は禁止する。
- 循環依存（A→B→Aのような閉路）は禁止する。
- 1つの後続タスクが複数の先行タスクを持つこと（合流構造）は許可する。この場合の日程計算は3.4.2の通り、すべての先行タスクの終了日のうち最も遅いものを基準とする。
- 依存関係は末端ノード（タスク）同士のみに設定できる。子を持つノード（グループ・プロジェクト）には設定できない。グループの表示期間はロールアップにより動的に変わるため、依存関係の基準として使うと結果が不安定になるためである。
- 開始日・終了日が未設定のノードは、依存関係の先行・後続いずれにも設定できない。
- 先行タスクの終了日と後続タスクの開始日が同日である場合は、整合が崩れているとはみなさない（後続は先行の終了日当日から開始できる）。
- 依存関係を新規に追加した時点で、その時点の日程に矛盾がないかを検証し、崩れていれば追加と同時に3.4.2のリスケジュールを適用する（日程変更時だけでなく、依存関係の追加時にも整合を保証する）。

## 3.8 ノードの削除（ソフトデリート）

- 親ノードを削除する場合、配下の子ノードもすべて連鎖的に削除扱いとする。
- 削除は論理削除（ソフトデリート）とする。削除されたノードは一覧・看板・ガント・リストいずれの画面からも表示されなくなるが、データそのものはスプレッドシート上に残る。
- 削除実行前に、削除される件数を示した確認ダイアログを表示する。
- 確認後に対象サブツリーが変化した場合は削除を中止し、最新件数で再確認する。
- 削除直後は通知トーストから元に戻せる。後から復旧する場合は、当面はスプレッドシートの変更履歴機能または管理者によるシートメンテナンスを利用する。
- 復元によって無効な依存関係や循環が再び有効になる場合は復元を拒否し、復元成功時は後続日程を最新値から再調整する。

## 3.9 通知連携

- プロジェクト設定で、Slack通知を「Slackアプリ方式」または「Workflow Builder方式」から選択する。Slackアプリ方式はIncoming Webhook（`/services/...`）を1本使用し、Workflow Builder方式はステータス変更用とコメントメンション用のWebhook（`/triggers/...`）を分けて使用する。
- Slackアプリ方式ではアプリが文面と装飾を組み立てる。Workflow Builder方式では用途別のフラットな変数をWebhookへ送り、本文・装飾・変数挿入はSlackのメッセージ送信ステップ側で設定する。単一の `{text}` 変数へ完成済み文面を渡す方式は新規設定では使用しない。
- 設定画面では、各方式を組織固有の申請事情ではなく文面・投稿先の管理場所の違いで説明する。Workflow Builder方式には用途別の作成手順、変数の追加順・型・値の例、本文への変数挿入例、URL保存と接続テストまでを表示する。
- ステータス変更通知にはタスク名、親パス、変更前後のステータス列、実行者名、WebアプリURLを利用できる。メンション通知にはタスク名、親パス、メンション対象、メンション対象者名、投稿者名、コメント本文、WebアプリURLを利用できる。
- Membersには任意のSlackメンバーIDを1人1件まで紐づけられる。同一SlackメンバーIDの重複登録は禁止する。Slackアプリ方式では登録済み対象者を `<@メンバーID>` として1件の通知へまとめ、未登録者は表示名だけを掲載する。Workflow Builder方式ではSlackメンバーID登録済みの対象者ごとに1件送信し、ユーザーID型変数を使って本人へ通知する。
- ステータス変更通知とコメントメンション通知は個別にON/OFFできる。Slackアプリ方式の各送信文面は許可されたプレースホルダーを使ってプロジェクト設定画面から編集・デフォルト復元でき、Workflow Builder方式の文面はSlack側で編集する。
- タスク追加、日程変更、カスケードリスケジュール、メンションを含まないコメントは通知対象外とする。
- Webhook URL が未設定または無効でも、タスク保存・コメント保存は失敗させない。
- Webhook URL、通知ON/OFF、送信文面、用途別の接続テスト、配信状態確認、方式別の連携解除はプロジェクト設定画面から操作できる。保存済みWebhook URL本体はブラウザへ返さないが、方式・用途ごとの設定済み状態と安全な固定マスク文字列を常時表示する。保存・接続テスト中は対象操作のボタンと補助表示にスピナーを出す。

## 3.10 マイルストーン・会議体

- プロジェクト設定画面から、WBS出力に使うマイルストーン（名称・日付・備考）と会議体（名称・開催ルール・備考）を管理できる。
- プロジェクト設定は広幅モーダルとし、左側にマイルストーン・会議体・Slack通知の縦タブ、右側に選択中の内容を表示する。
- マイルストーンと会議体はアプリ内ガントビューの予定レーンと、WBS出力上部の一覧・日付グリッドに反映する。

## 3.11 WBS出力

- トップバーから `WBS` シートを上書き生成できる。アプリは `WBS` シートを読み込まず、シート上の手編集は次回出力で消える。
- 出力対象は `IncludeInWbs` が FALSE でないノードとし、FALSE のノードは子孫ごと除外する。採番・階層列・日付範囲は除外後のツリーに対して計算する。
- WBSには成果物、備考、会社名、責任者、計画日、実績日、進捗率、完了フラグ、計画ガント、実績マーカー、マイルストーン、会議体を出力する。
- 実績日はNodesの手動実績期間を優先し、未設定ならActivityLogから自動算出する。実績期間が計画期間外にある場合も、400日の出力上限内で日付グリッドの範囲に含める。
- 未同期の保存がある間は、シートに未到達の変更が漏れないようWBS出力を実行できない。
- WBSは一時シートへ完成させてから切り替え、途中失敗時は直前のWBSを残す。同時出力はサーバー側でも拒否する。
- 名称確定前の作成途中ノード（DraftOwnerが設定されたノード）は、WBS番号・期間・進捗集計を含む出力対象から除外する。

# 4\. 非機能要件

## 4.1 UI/UXデザイン方針

- モダンで洗練されたビジュアルデザインとする。
- 説明文的な補助テキストは極力排除し、必要最小限の情報のみを表示するシンプルな構成とする。
- 保存確定に体感の遅延が生じうるため（4.2参照）、操作直後にUI上は即座に反映し、裏側で保存する「楽観的更新」を基本とする。ただし操作が連続する場合の順序保証・古い応答の破棄については、設計側でクライアント保存キューとして仕組み化する（設計書4.7）。

## 4.2 パフォーマンスとレイテンシ

- ノード総数〜100件程度を前提に、起動時に全件をロードしてクライアント側で保持・描画する。
- GASの1回あたりの実行時間上限（6分）は本規模では問題にならない。実運用上の主なリスクは google.script.run 呼び出しごとの往復遅延（体感1〜数秒）であり、保存確定操作のたびにこの遅延が発生する前提でUI設計を行う（4.1の楽観的更新方針）。
- 差分同期・仮想スクロール等の大規模データ対応は本バージョンのスコープ外とする。

## 4.3 同時編集・競合方針

- 他ユーザーの変更をリアルタイムに画面反映する機能は設けない。基本的には1人ずつ操作する運用とし、まれに操作が重なった場合のみ保存時に警告する。
- 保存時、対象ノードの最終更新日時を照合し、自分が読み込んだ後に他ユーザーが更新していた場合は上書き保存を防ぎ、警告を表示する。
- 書き込み処理はLockServiceによる排他制御を行い、同時書き込みによるデータ破損を防ぐ。

## 4.4 本人識別・認証

- ログイン中のGoogle Workspaceアカウントのメールアドレスをもとに、メンバー一覧と照合して本人を自動識別する。
- 未登録のメールアドレスの場合は、参加モーダルからセルフ登録できる。登録せず「あとで」を選んだ場合は、コメント投稿・更新者記録を伴う保存操作は利用できない旨を案内する。
- ただし、初回セットアップ（メンバー0件の状態での初期化）とセルフ登録（セッション本人のメールアドレスのみ登録可能）に限り、この本人識別チェックの対象外とする。

## 4.5 アクセス権限

- 利用者は全員が閲覧・編集可能とし、機能・データ単位での権限制御は行わない。

## 4.6 データ永続化・日付形式

- すべてのデータはコンテナバインドされたスプレッドシート上に保存する。
- 日付はタイムゾーンに起因する不整合を避けるため、クライアント・サーバー間およびシート上の保存形式を「YYYY-MM-DD」のISO文字列に統一する。

# 5\. スコープ外（対象外）

- ユーザーごとの権限管理・アクセス制限
- 複数案件をまたぐデータ連携・案件切り替えUI（1スプレッドシート=1案件のため）
- リアルタイム同期（他ユーザーの操作の即時反映）
- モバイル専用に最適化されたUI（PCでの利用を主眼とする）
- 変更履歴の完全な監査ログ・Undo/Redo機能
- プロジェクトテンプレート機能、アーカイブ機能
- 101件以上の大規模データに対応するための性能最適化

# 6\. オープン事項（実装前に要確認）

- 説明欄・コメント欄の文字数上限。
- 削除確認・競合警告など、UI文言の最終確定。
- PC利用を前提とした場合の最小対応画面幅。
- 依存関係の追加・編集を、どの画面（ガント上のドラッグ接続／詳細モーダルからの選択）から行うかのUI詳細。
- ソフトデリートされたノードを一覧から探して復元する管理者向け機能を将来的に追加するかどうか（現状は削除直後のUndoのみアプリ内で提供する）。
