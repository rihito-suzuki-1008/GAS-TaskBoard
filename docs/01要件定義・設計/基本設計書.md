# 1\. システム構成

## 1.1 全体アーキテクチャ

コンテナバインドされたGASプロジェクトが、doGet(e) を通じて独立したWebアプリのHTMLを配信する。フロントエンドはHTML Service上で動作するJavaScriptとし、データの読み書きはgoogle.script.run経由でGASのサーバーサイド関数を呼び出す。データの実体はスプレッドシートの各シートに保存する。

［ブラウザ］

　└ HTML Service（フロントエンド：看板／ガント／リスト／モーダル／メンバー管理）

　　└ google.script.run （非同期呼び出し・往復ごとに遅延あり）

　　　└ ［GASサーバーサイド関数］

　　　　└ SpreadsheetApp（Nodes / Members / StatusColumns / Dependencies / Comments）

## 1.2 デプロイ設定（本人識別の前提）

実行モードは「アクセスしたユーザーとして実行」を採用する。これは意識的なトレードオフの選択であり、理由と代償を以下に記す。

| **設定項目** | **値** | **理由** |
| --- | --- | --- |
| 実行するユーザー | アクセスしたユーザー | この設定でも「自分（オーナー）として実行」でも、同一Google Workspaceドメイン内であれば Session.getActiveUser().getEmail() でアクセス本人のメールアドレスを取得できる（本人識別はどちらの実行モードでも成立する）。本アプリで「アクセスしたユーザーとして実行」を選んだ理由は、書き込みの実行権限を各利用者自身に持たせ、特定の1アカウント（オーナー）に依存しない運用にするためである。 |
| アプリケーションにアクセスできるユーザー | 組織内の全員（同一Google Workspaceドメイン） | 個人Gmailアカウントの混在を許すと、getActiveUser() が空文字を返しうるため、ドメイン限定を必須とする。 |
| スプレッドシートの共有設定 | 全利用者にコンテナスプレッドシートの編集権限を付与する | 「アクセスしたユーザーとして実行」では各利用者自身の権限で書き込みが行われるため、編集権限がないユーザーは本人識別以前の権限エラーで保存に失敗する。 |

_※ この設定の代償として、利用者はコンテナスプレッドシートを直接開いて編集することも可能であり、看板の完了列制約・依存関係の循環禁止・本人識別などアプリ側のバリデーションは、直接編集によって迂回されうる。この点は社内の少人数チーム利用（性善説が成立する運用）を前提に許容している。直接編集を技術的に禁止したい場合は「自分（オーナー）として実行」への変更を検討する（その場合、利用者へのシート編集権限付与が不要になる代わりに、書き込みはすべてオーナーのアカウント権限・クォータで実行され、オーナーの退職・異動時に再デプロイが必要になる）。_

_※ Membersは担当者候補と保存主体を識別するためのアプリ内マスタであり、アクセス制御リストではない。スプレッドシート編集者は、コンテナバインドされたApps Scriptのコードを閲覧・変更でき、Script Propertiesを読み出すコードも実行できる。この構成では全編集者を信頼済みの管理主体として扱う。案件離脱・退職時はMembers削除だけで済ませず、Driveまたは共有Google Groupの権限を運用管理者が削除する。Webhook等の厳密な秘密分離や管理者ロールが必要になった場合は、現行の権限モデルのまま機能追加せず、別backendを含めて再設計する。_

_※ ドメイン外アカウントが混在する運用に変わる場合は、本人識別の設計全体を見直す必要がある。_

## 1.3 技術構成

| **レイヤー** | **技術・方式** |
| --- | --- |
| 配信方式 | doGet(e) によるWebアプリ公開 |
| フロントエンド | Vanilla JavaScript を基本方針とする。GAS上での保守性・依存ライブラリ管理の簡素化・デプロイ容易性を優先し、重量フレームワークの導入は避ける。必要に応じてHTTPS経由の軽量ライブラリ（日付操作等）のみ許容する。 |

# 2\. データモデル設計

## 2.1 シート構成一覧

| Members | 担当者・投稿者候補のマスタ（Emailで本人と紐づく） |
| StatusColumns | 看板ビューのステータス列定義（順序・完了フラグを含む） |
| Dependencies | ノード間の先行・後続（依存）関係 |

| Milestones | WBS出力用マイルストーン |
| Meetings | WBS出力用会議体 |

_※ グループ／タスクの区別は列を持たず、他ノードから ParentId で参照されているかどうかにより実行時に判定する。_

## 2.2 Nodes シート

| **列名** | **型** | **説明** |
| --- | --- | --- |
| NodeId | 文字列（UUID） | 一意識別子。 |
| ParentId | 文字列（UUID） | 親ノードの NodeId。ルート（プロジェクト）のみ空欄。ルートは常に1件。 |
| Name | 文字列 | ノード名。必須。 |
| StatusColumnId | 文字列 | StatusColumns シートの ColumnId を参照。看板上の位置付けにのみ使用（進捗計算には子を持つノードでは使用しない）。 |
| AssigneeIds | 文字列（カンマ区切り） | Members シートの MemberId を複数格納。メンバー削除時に自動で除去される。 |
| Priority | 列挙（High/Mid/Low） | 優先度。 |
| Tags | 文字列（カンマ区切り） | 廃止済み（4.1）。アプリは読み書きしない。既存シートの列位置がずれないよう、列定義自体は残している。 |
| StartDate / EndDate | 文字列（YYYY-MM-DD、任意） | ユーザーが明示的に設定した期間。未入力を許可する。ネイティブDate型は使用しない。未設定の場合、ロールアップ計算（4.2）から除外し、依存関係（2.5）の先行・後続には設定できない。 |
| ActualStartDate / ActualEndDate | 文字列（YYYY-MM-DD、任意） | WBS実績期間の手動上書き。既存列を動かさずNodes末尾へ追加する。両方設定または両方空欄だけを許可し、空欄時はActivityLogの自動算出へフォールバックする。計画期間・依存関係・ロールアップには使用しない。 |
| Description | 文字列（長文） | 説明・詳細。 |
| SortOrder | 数値  | 同一階層内での並び順。1000刻みなど間隔をあけて採番し、挿入時は中間値を採番する（全件再採番を避ける）。 |
| CreatedAt / UpdatedAt | 文字列（ISO日時） | 監査・競合検知用。 |
| UpdatedBy | 文字列 | 最終更新者の MemberId（2.7参照）。 |
| DeletedAt | 文字列（ISO日時、任意） | 論理削除日時。空欄なら未削除。値が入っている行は全画面・全API応答から除外する（2.10参照）。 |
| DeletedBy | 文字列（任意） | 削除操作を行った MemberId。 |
| Deliverable | 文字列（任意） | WBS出力用の成果物。 |
| Note | 文字列（任意） | WBS出力用の短い備考。Descriptionとは用途を分ける。 |
| Progress | 数値（任意） | 末端ノードの手動進捗。0/15/30/45/60/75/90/100のみ許可。 |
| IncludeInWbs | 真偽値（任意） | WBS出力対象フラグ。空欄はTRUEとして扱う。FALSEはサブツリーごと除外。 |
| DraftOwner | 文字列（任意） | 名称確定前の作成途中ノードを所有するMemberId。通常ノードは空欄。作成途中ノードは所有者本人だけに返す。 |
| DraftExpiresAt | 文字列（ISO日時、任意） | 作成途中ノードの自動整理期限。通常ノードは空欄。期限切れ時はソフトデリートする。 |

_※ Nameは原則必須だが、作成UIの継続性のためDraftOwnerが設定された作成途中ノードだけは一時的な空欄を許可する。名称保存時にDraftOwner / DraftExpiresAtを消去する。_

_※ 親ノードのProgress（進捗率）およびガント上の「拡張表示期間」は Nodes シートに保存しない。末端ノードのProgressはユーザー入力値として保存し、親は読み込み・保存のたびにサーバーサイドで動的に算出する。理由は2.9を参照。_

## 2.3 Members シート

| **列名** | **型** | **説明** |
| --- | --- | --- |
| MemberId | 文字列 | 一意識別子。 |
| Name | 文字列 | 表示名。 |
| Email | 文字列 | Google Workspaceアカウントのメールアドレス。Session.getActiveUser().getEmail() との照合キー。保存時に前後の空白をトリムし、小文字に正規化したうえで、重複を禁止する。 |
| Color | 文字列（任意） | UI上のアバター表示色。 |
| Company | 文字列（任意） | WBS出力で担当者の会社名として表示する。 |
| SlackUserId | 文字列（任意） | Slackの固定メンバーID（通常はU、Enterprise環境ではWから始まる英数字）。コメント通知の本人メンションに使用する。前後空白を除去して大文字に正規化し、Members内の重複を禁止する。 |

_※ メンバーが0件になると本人識別が成立せずアプリ全体が保存不能になるため、Membersシートの最後の1行を削除する操作はサーバー側で拒否する（要件定義書3.3）。_

## 2.4 StatusColumns シート

| **列名** | **型** | **説明** |
| --- | --- | --- |
| ColumnId | 文字列 | 一意識別子。 |
| Name | 文字列 | 列名。ユーザーが自由に編集可能。 |
| SortOrder | 数値  | 看板ビューでの列の表示順。 |
| IsDoneColumn | 真偽値 | 完了列フラグ。常に厳密に1件のみtrueとなるようサーバー側で強制する（trueに設定した瞬間、他の全行を自動でfalseにする）。IsDoneColumn=trueの列は、所属ノードの有無にかかわらず削除できない。削除するには、先に別の列を完了列に指定し（自動的に旧完了列はfalseになる）、そのうえで削除する。 |
| IsInProgressColumn | 真偽値（任意） | 親ステータスの自動ロールアップで使う進行中列。明示設定後は列名を変更しても役割を維持する。既存シートで空欄の場合は「進行中」等の従来名から互換判定する。完了列と同じ列には設定できない。 |

完了列の切り替えはStatusColumns全行を1回の `setValues` で更新し、0件または複数件trueの中間状態を作らない。読み込み時も厳密に1件であることを検証し、不正時に先頭列などへフォールバックしない。

完了列または進行中列の役割を切り替える場合は、同じロック内で新しい役割を基準に末端ノードのProgressを正規化し、親ノードのStatusColumnIdを深い階層から順に再計算する。クライアントへは派生進捗を含む全有効ノードを返し、列定義とノード表示を1応答で収束させる。

| **列名** | **型** | **説明** |
| --- | --- | --- |
| DependencyId | 文字列 | 一意識別子。 |
| PredecessorNodeId | 文字列 | 先行タスクの NodeId。 |
| SuccessorNodeId | 文字列 | 後続タスクの NodeId。 |

## 2.6b ActivityLog / Milestones / Meetings シート

ActivityLog はステータス変更・進捗変更のみを追記する履歴シートで、実績開始日・実績終了日の自動算出に使う。列は LogId, NodeId, Field, OldValue, NewValue, NewValueIsDone, ChangedAt, ChangedBy とする。日程カスケードや名称変更、追加、削除は記録しない。詳細ドロワーでは開いたノードの算出値だけを遅延取得し、通常の全体読み込みへActivityLog全件を含めない。

Milestones は MilestoneId, Name, Date, Note, SortOrder、Meetings は MeetingId, Name, Schedule, Note, SortOrder, ScheduleRuleJson, StartDate, EndDate とする。Meetings の Schedule は表示ラベル、ScheduleRuleJson と StartDate / EndDate は毎日・毎週・隔週・月次開催の展開用データである。いずれも LockService による直列化のみで、UpdatedAt照合は追加しない。

## 2.6c CalendarOverrides シート

CalendarOverrides は案件共通カレンダーの手動例外だけを保存する。列は Date, DayType, Name とし、Date は YYYY-MM-DD、DayType は `working`（稼働日）または `holiday`（休日）、Name は任意名称とする。Date は一意で、同一日を複数行にしない。「標準」は行として保存せず、既存の例外行を物理削除して土日・日本の祝日からの自動判定へ戻す。

このシートはガントの背景表示だけに使用する。タスクのStartDate / EndDate、期間日数、依存関係の制約判定・自動リスケジュール、親のロールアップ期間、WBS出力には使用しない。保存は本人識別後に LockService で直列化し、NodesのUpdatedAtによる楽観的競合検知は適用しない。

## 2.6d NotificationReads シート

NotificationReadsは、CommentsのMentionsから動的に構成するアプリ内通知へメンバー別の既読状態を重ねるための疎な既読記録である。列は NotificationKey, RecipientMemberId, NotificationType, SourceId, ReadAt とする。

- NotificationKeyは `mention:<CommentId>:<MemberId>` の決定的な文字列とし、同じ通知の既読行を重複作成しない。
- RecipientMemberIdは通知を受け取ったMembers.MemberId、NotificationTypeは当面 `mention`、SourceIdはComments.CommentId、ReadAtはISO日時文字列とする。
- 未読通知は行として保存しない。Comments.Mentionsに現在メンバーを含み、対応するNotificationReads行が存在しないコメントを未読と判定する。これにより機能追加前のコメントも通知一覧へ含める。
- 既読化は本人識別とLockServiceで直列化し、対象コメントが実際に現在メンバーをメンションしていることをサーバーで再検証する。NodesのUpdatedAtによる競合検知は適用しない。
- NotificationReadsは通知カードの表示内容を複製しない。投稿者名・本文・日時はComments、タスク名・階層は有効なNodesから参照する。論理削除ノードは通知対象から除外する。

## 2.7 本人識別ロジック

- サーバー関数の冒頭で Session.getActiveUser().getEmail() を取得する。
- 取得したEmailで Members シートを検索し、一致する MemberId を本人として確定する。
- 一致するメンバーが存在しない場合、通常のコメント投稿・保存操作は拒否し、メンバー未登録である旨を返す。クライアントは参加モーダルからセルフ登録へ誘導できる。
- この識別チェックは、通常のノード編集・コメント投稿・メンバー編集など、Membersシートが既に存在するあらゆる保存操作に適用する。例外は2.11の初回セットアップ処理と、セルフ登録API `joinAsMember` の2つだけとする。`joinAsMember` は `Session.getActiveUser().getEmail()` で取得したセッション本人のメールアドレスだけを登録し、payload由来のEmailは受け取らない。
- 時間主導型トリガーの `scheduledExportWbs` は無人実行でActiveUserを前提にできないため、`Session.getEffectiveUser().getEmail()` でトリガー作成者を取得し、Members.Emailとの一致を必須とする。これは本人識別を省略する例外ではなく、インストール型トリガー専用の実行者識別である。通常の保存APIへEffectiveUser方式を流用しない。

## 2.8 日付・日時列のフォーマット

4.6（要件定義書）でStartDate・EndDate等をISO文字列（YYYY-MM-DD）で統一する方針としたが、Googleスプレッドシートは既定の表示形式のままだとセルへの文字列書き込みを日付値へ自動変換してしまうことがある。この自動変換を防ぐため、Nodesシートの StartDate・EndDate・ActualStartDate・ActualEndDate・CreatedAt・UpdatedAt・DraftExpiresAt、Commentsシートの Timestamp / ParentCommentId / Mentions列、CalendarOverridesシートの Date / DayType / Name列、NotificationReadsの全列は、シート初期化時に表示形式を「書式なしテキスト」（setNumberFormat("@")）に設定し、常に文字列として保持・取得されることを保証する。

スキーマ自動移行は既知の末尾列追加だけを許可する。列の並び替え、途中列の欠落、未知列を検出した場合はヘッダーを上書きせず `SCHEMA_MISMATCH` で停止する。

## 2.9 ロールアップを「保存しない」設計にした理由

初版では、子ノードの日付・進捗を親ノードの行に書き込んで保存する設計だった。この場合、複数ユーザーが同じ親を持つ別々の子ノードを同時に編集すると、双方の処理が同じ親の行を上書きし合い、後勝ちで片方の更新が消える競合が発生しうる。

本版では、進捗率と同様、親ノードの「拡張された表示期間」も保存せず、読み込み・保存のたびにサーバーサイドで子孫ノードから動的に計算する方式に統一した。Nodesシートに実際に保存される期間は、ユーザーが手動で設定した値、または「子に合わせる」ボタン押下時の値のみである。これにより、子ノードの編集が親ノードの行への書き込みを発生させなくなり、上記の競合が構造的に起きなくなる。同じ考え方を4.3の依存関係リスケジュールにも適用している。

## 2.10 ソフトデリート設計

ノードの削除は物理削除ではなく論理削除（ソフトデリート）とする。GAS＋スプレッドシートにはトランザクション機構がなく、削除処理の途中で失敗した場合に元へ戻す手段がないため、実際の行を消さない方が安全である。

- 削除操作は、対象ノードおよびその子孫すべてのDeletedAt・DeletedByに値を設定するだけであり、行自体は残す。Comments・Dependenciesの関連行も削除しない。
- 削除直後のUndoでは、対象ノードおよび子孫のDeletedAt・DeletedByを空に戻し、UpdatedAt・UpdatedByを更新する。
- Undo前に、復元によって有効になるDependenciesを末端ノード・日付・重複・DAGの各条件で再検証する。無効な依存関係が復活する場合は復元を拒否する。復元成功時は最新シート値を基準に後続タスクを再スケジュールする。
- loadAll等の読み込み系API・各画面表示は、DeletedAtが空でないノードを結果から除外する。DeletedAtが設定されたノードに紐づくComments・Dependenciesも、表示・計算の対象から除外する（対象ノードが表示されないため実質的に連動して隠れる）。
- loadAllは有効ノードについてNodeId重複、ルート件数、親なし孤児、階層循環を検証し、不整合時は通常表示を停止する。
- 完全な物理削除（スプレッドシートから行そのものを消す作業）は、必要になった場合のみ管理者がスプレッドシートを直接操作するメンテナンス作業として行う。アプリの通常機能としては提供しない。

## 2.11 初期セットアップフロー

案件用にスプレッドシートをコピーした直後は、Nodes・Members・StatusColumnsのすべてが空である。この状態でも起動直後からアプリとして機能するよう、初回セットアップで以下を行う（2.7の通り、本人識別チェックの対象外とする）。処理は冪等にし、途中失敗でMembersまたはStatusColumnsだけが作成済みの場合は既存行を再利用して不足分から再開する。ルートノードの作成をセットアップ完了のコミットポイントとする。

- 案件名の入力を求め、ルートノード（ParentId空欄）を1件作成する。
- 初期ステータス列として「未着手」「進行中」「完了」の3列を自動生成し、「完了」列のIsDoneColumnをtrueにする。
- Session.getActiveUser().getEmail() で取得したアクセス本人のメールアドレスをもとに、そのユーザーを最初のMembersレコードとして自動登録する。
- これら3つの処理は1回のセットアップ操作としてまとめて行い、一部だけが実行された中途半端な状態を残さない。

_※ 「メンバー管理は全員が行える」という3.3・4.5の方針上、最初のメンバーに特別な管理者権限は与えない。単に自動登録されるだけである。_

# 3\. 画面設計

## 3.1 共通レイアウト

- 画面上部：案件名表示（切り替えUIなし）／ビュー切り替えタブ（看板・ガント・リスト）／同期状態と手動最新確認を兼ねる同期ピル／自分宛メンション通知のベル／メンバー管理・プロジェクト設定・WBS出力への導線。WBS出力は低頻度の上書き操作としてSecondary表示にする。
- プロジェクト設定：最大幅約1120pxのモーダル内を「左側の縦タブ／右側の設定内容」に分ける。左タブはマイルストーン・会議体・Slack通知を持ち、960px以下では内容幅を優先し、720px以下では上部の横スクロールタブへ切り替える。
- 共通フィルタバー：既定表示はキーワード検索・「自分のタスク」・「＋フィルタ」とし、階層・ステータス・担当者・優先度・期間は「＋フィルタ」から追加する。適用中の条件は再編集・個別解除できるチップだけで表示し、常設ドロップダウンとの二重表示は行わない。
- 画面主要領域：選択中のビューを表示。
- ノードクリックで右側の詳細ドロワーを表示。

## 3.2 看板ビュー

- 横方向にステータス列、列内に縦積みでカードを表示。
- カード表示項目：名称、担当者アイコン、優先度、期限（期限超過は強調色）。
- カードを別列へドラッグ&ドロップすると、詳細ドロワーを開かずにステータス（StatusColumnId）を変更できる。ローカル表示を即時更新し、保存は裏側で行う楽観的更新とする（4.7参照）。
- 列の追加は画面上部の「＋列追加」ボタンから行う。列ヘッダー自体は「ドラッグハンドル＋列名＋（完了列なら）完了列チップ＋件数バッジ＋タスク追加アイコン＋列編集アイコン」のみの1行構成とし、複数の操作ボタンを列ヘッダーに並べない（視覚的な乱雑さを避けるため）。
- タスクの追加は、空の列に表示される「＋」領域のクリック、または各列ヘッダーのタスク追加アイコンから行う（列が空でなくてもその場で追加できる）。画面上部の共通「＋タスクを追加」ボタンは置かない。
- 並べ替えは列ヘッダーをドラッグして行う。ドラッグ中はドロップ先候補列の左端または右端に挿入位置を示すガイド（縦バー）を表示し、ドロップした位置へ挿入する。並び順はドロップした瞬間にブラウザ上で即時反映し、スプレッドシートへの書き込みはバックグラウンドで行う（失敗時のみ元の順序へ戻す）。
- 列名編集・色変更・完了列指定・列削除は、列ヘッダーの編集アイコンから開く編集モーダルに集約する。完了列の指定はモーダル内のトグルスイッチ（チェックボックス）で行い、保存時に他列のOFFをサーバー側で強制し画面にも反映する。列削除は編集モーダル内のDestructiveボタンから確認モーダルを経由して行い、所属ノードがあればエラーを表示し中止する。列ヘッダーに削除アイコンは常設しない。
- 画面上部トグルで「末端ノードのみ表示 / 親ノードも表示」を切り替え。
- 列内のカード順は、フィルタ一致ノードをリストと同じ深さ優先の階層順へ重ねて決める。ステータス列やフィルタにより非表示ノードがあっても、表示カード同士の相対順を維持する。
- 論理削除（DeletedAtが設定された）ノードはカードとして表示しない（2.10参照）。
- 列本文は縦スクロール領域とし、自動生成するカード行を `max-content` で積む。カード数が多い場合も列の残り高さへ均等圧縮せず、各カードはタスク名が最低1行読める高さを維持する。

## 3.3 ガントビュー

- 左側：階層ツリー（折りたたみ可能。横スクロール時も固定表示）。ツリーの名称セルにはリストビューと同様に編集アイコンを常設し、ホバーで背景色が変わることでクリック可能であることを示す。右側：日付軸に沿ったバー表示領域。
- タスク行の上部にマイルストーン/会議体レーンを表示する。レーンは折りたたみ可能で、マイルストーンはラベル付き、会議体は開催日の `▼` で表示し、ホバーで内容を確認できる。
- バー中央のドラッグで期間移動、バー左右端のドラッグで開始日／終了日を個別に変更。ドラッグ中はローカル表示のみを即時更新し、確定（mouseup）時にサーバーへ保存する。
- 保存時、依存関係で連鎖する後続ノード群の日程を再計算し、影響を受けたノードをまとめてクライアントへ返却、画面に反映する（5章のAPI契約を参照）。
- 画面上部にズーム切り替え（日／週／月）を配置。ズームは表示上の粒度のみを変更し、データそのものは日単位で保持する。
- 日表示では土曜日を水色、日曜日および日本の国民の祝日・振替休日・国民の休日をピンクで示す。祝日と土曜日が重なる場合は祝日扱いのピンクとする。今日は曜日・休日より優先して1日分のセル幅全体をオレンジ系の背景色で示し、色を混ぜない。
- 日付ヘッダーの各日をクリックすると案件カレンダー編集モーダルを開く。開始日・終了日で期間を指定し、「標準」「稼働日」「休日」と任意名称を設定できる。標準は手動例外を削除する。CalendarOverridesの手動設定を既定の土日・祝日判定より優先するが、今日のオレンジ表示はさらに優先する。
- 案件カレンダーは表示上の区別だけとし、予定日、期間日数、依存関係の日程計算、WBSには影響させない。
- 親のバーが子孫の範囲を包含していない場合は表示上自動拡張。親のバーが子孫の範囲より広い場合は「子に合わせる」ボタンを表示。
- 依存関係は先行タスクの終端から後続タスクの始端へ矢印線（SVG）で接続する。経路は先行バー右端から外へ出し、対象行間で最も右まで伸びるバーの外側に確保したレーンを通して、後続バー左端へ入る直角折れ線とする。重なる行区間の経路には別レーンを割り当て、線には背景色の縁取りを描いて交差を判別できるようにする。折りたたみ中の親子をまたぐ依存は折りたたまれた親ノードへアンカーし、同一の表示ノード間へ集約された複数依存は1本にまとめ、同じ表示ノード内に閉じる依存は描画しない。
- 開始日・終了日が未設定のノードはバーを描画しない（ツリー上には表示するが、タイムライン領域には表示しない）。

_※ ガントチャートは本アプリで最も実装リスクが高い機能である。看板・リスト・詳細モーダル・メンバー管理が一通り完成した後に着手することを推奨する。実装順序の目安：①バー表示のみ（読み取り専用）→②バーのドラッグ移動・リサイズ→③依存関係の矢印描画→④依存関係の自動連鎖リスケジュール、の順に段階的に検証しながら進めることで、手戻りのリスクを抑えられる。_

## 3.4 リストビュー

- インデントでツリー構造を表現した表形式。列：名称／ステータス／担当者／優先度／期間／進捗率。
- 名称、ステータス、担当者、優先度、期間、末端タスクの進捗率は行内で直接編集できる。通常時はチップや日付バッジとして見せ、クリック後だけ入力UIを開く。
- 行左のドラッグハンドルで表示順と親子関係を変更できる。親タスクを移動すると子孫も一緒に移動する。
- 列ヘッダーでのソートボタンは置かず、表示順はSortOrderとD&Dを正とする。行の折りたたみに対応。

## 3.5 タスク詳細ドロワー

- デスクトップでは最大約1152pxの広幅ドロワーとし、左の主領域と右の属性領域へ分ける。960px以下では1列に戻す。
- 左の主領域：案件ルートを除いた `親 > 子 > 現在タスク` の階層補助表示、大きな名称入力、説明欄、コメントスレッド、新規コメント入力。説明とコメントが属性フォームより広い表示領域を持つ。
- 右の属性領域：ステータス、担当者、優先度、進捗、予定期間、実績期間、WBS項目、親子ナビゲーション、依存関係、削除操作。
- 実績期間が空欄ならActivityLog由来で自動算出し、手動期間を設定した場合はその値をWBSへ優先反映する。「クリア」は両方の手動実績日を空にして自動へ戻す。詳細ドロワーの補助文は手動・自動・遅延取得中で切り替えず、空欄時の扱いを示す固定文言にする。

- 通知カードから開いた場合は通常の最新コメントページとは別に対象コメントとその親スレッドを直接取得し、対象コメントへスクロールして短時間強調する。取得・表示に成功した後で当該通知を既読化する。
- 進捗率は末端タスクのみ8段階で編集可能。親タスクは子タスクから算出された値を読み取り専用で表示する。
- 通常保存はボタン押下直後にドロワーを閉じ、保存処理はバックグラウンドで継続する。

- 削除時は、当該メンバーが担当者として登録されている全ノード数を確認表示したうえで実行する。
- 操作は全利用者に開放し、管理者専用の制限は設けない（2.10・4.5参照）。

# 4\. 主要ロジック設計

## 4.1 進捗率算出ロジック

- 末端ノード：StatusColumnId が IsDoneColumn=true の列であれば Progress=100、それ以外は Nodes.Progress が有効値ならその値、未設定なら0。
- saveNodeでは「完了列 ⇔ Progress=100」を正規化する。進捗100%指定は完了列へ移動し、完了列から他列へ戻した場合はProgress=90へ戻す。親ノードへの進捗パッチは拒否する。
- 子を持つノード：直下の子ノードの Progress を再帰的に算出し単純平均する（自身の StatusColumnId は無視する）。日付を持たない子や論理削除された子は計算対象から除外しない（進捗計算は日付の有無に依存しない）。
- 親ノードのStatusColumnIdは、子の状態に合わせて保存時に補正する。直属の子がすべて完了列なら親も完了列へ、子に進行中列がある場合、または全子完了ではないが完了列の子が1つでもある場合は親を進行中列へ寄せる。
- 算出はノード読み込み時・保存応答時にサーバーサイドで行う。あるノードのステータスを変更した場合、そのノードの祖先すべての進捗率が変化しうるため、保存応答には変更対象ノードだけでなく、祖先チェーン全体の再算出後の値を含める（5章API設計参照）。

## 4.2 ガント：期間の動的ロールアップと「子に合わせる」

- 表示用の期間は「自身のStartDate/EndDate」と「子孫のバウンディング範囲（最小開始日・最大終了日を再帰計算）」を比較し、大きい方を都度算出してクライアントに返す。この値はNodesシートには書き込まない。
- バウンディング範囲の計算では、StartDate・EndDateが未設定の子孫は計算対象から除外する（存在しないものとして扱う）。子孫がすべて日付未設定の場合、親の表示期間は自身の手動設定値のみとなる。
- 「子に合わせる」ボタン押下時のみ、その時点の子孫バウンディング範囲を自身のStartDate/EndDateとして実際に書き込む。

## 4.3 依存関係の自動連鎖リスケジュール

- 先行タスクの日程変更を保存する際、依存関係グラフ上でその変更が影響しうるノード群（後続タスク、およびその後続…と連鎖するすべて）を特定する。
- 特定したノード群を、依存関係の向き（先行→後続）に沿ったトポロジカル順に並べ、その順番で1件ずつ整合チェックと繰り下げを適用する。ある後続ノードが複数の先行タスクを持つ場合（合流構造）は、そのノードを処理する時点で既に計算済みの「すべての先行タスクの終了日のうち最も遅いもの」を基準に判定する。こうすることで、一本道の依存だけでなく合流のある依存グラフでも正しく計算できる。
- 整合チェックは「後続ノードのStartDateが、基準となる先行タスクの終了日より前になっていないか」のみを見る。崩れている場合は、崩れた分だけ後続ノードのStartDate・EndDateを繰り下げる（元の期間の長さは保つ）。崩れていない場合（ユーザーが意図的に空けたバッファがある場合）は何もしない。常に隣接させるスナップ処理は行わない。
- この計算はLockServiceのロックを取得した後、対象ノード群の値をシートから改めて読み直したうえで行う。クライアントが保持するキャッシュ値は一切使用しない。これにより、他ユーザーが直前に確定させた変更（後続ノードの手動編集を含む）を必ず起点として計算するため、後から実行された保存が古い情報をもとに他ユーザーの更新を上書きしてしまう事故（ロストアップデート）が構造的に起きない（2.9のロールアップと同じ考え方）。
- 合流構造（複数の先行タスクを持つノード）を判定する際は、依存関係の連鎖（フォワードクロージャ）に含まれない先行タスクについても、その時点の最新値をシートから読み直して基準に含める。変更の起点から見て「連鎖の外」にいる先行タスクであっても、判定対象ノードにとっては直接の先行であるため、その値を無視すると合流判定を誤る。
- 繰り下げによって書き換えたすべての行（先行ノード自身＋連鎖的に変更した後続ノード群）は、StartDate・EndDateだけでなく UpdatedAt・UpdatedBy も同時に更新する。これにより、この保存より後に発生する別ユーザーの保存操作が、書き換え済みの最新状態を正しく基準として競合検知できる（4.6参照）。
- 影響を受けたノード群（先行ノード自身＋連鎖的に繰り下げられた後続ノード群）をまとめて1回の書き込みで保存し、保存応答としてすべての変更後ノードをクライアントへ返す。
- クライアントは応答に含まれる「影響を受けたノード数」をもとに「この変更により、後続タスクN件の日程を自動調整しました」という趣旨の通知を表示する。
- このリスケジュール処理は、既存ノードの日程変更時だけでなく、addDependencyで新たな依存関係を追加した直後にも同じロジックで呼び出す（2.5参照）。追加した時点で既に整合が崩れているケースに対応するためである。

## 4.4 ドラッグ&ドロップによる階層付け替え

- ツリー上でノードをドラッグし、別ノード上にドロップすると ParentId を更新する。
- 自分自身の子孫を親に設定しようとする操作はサーバーサイドで検知し拒否する（循環防止）。

## 4.5 ノードの削除（ソフトデリート）

- 削除対象ノードの子孫を再帰的に収集し、件数を確認ダイアログで提示したうえで、対象ノードと収集した子孫すべての DeletedAt・DeletedBy に値を設定する（2.10参照）。行の削除やComments・Dependenciesへの変更は行わない。
- 確認時の対象NodeId一覧と基準UpdatedAtをサーバーへ渡す。ロック取得後の最新サブツリーと一致しない場合は削除せず、最新件数での再確認を求める。
- 対象サブツリーとロールアップ対象行は、行番号が連続するブロックごとに `setValues` で書き込む。離れた対象行の間にある無関係行はread-modify-writeせず、直接シート編集や数式セルを巻き込まない。
- 削除成功後はトーストにUndoアクションを出し、`restoreNode` で同じサブツリーを復元できる。親タスク自体が削除済みの場合、その配下だけの復元は拒否する。

## 4.6 排他制御・競合検知

- 書き込み処理は LockService.getScriptLock() で保護する。ロック取得の待機上限は15秒とし、取得できない場合は構造化された `LOCK_BUSY` を返す。クライアントは書き込み前に確定したこのエラーだけを指数バックオフ付きで最大2回試行する。
- ロック取得後は try/finally で確実に解放し、解放前に SpreadsheetApp.flush() を呼び、書き込みを確定させてから次のロック取得者に処理を渡す。
- 楽観的な競合検知は、ユーザーが直接編集した対象ノード1件についてのみ、保存時に UpdatedAt を照合する（クライアントが読み込んだ時点の値と現在の値を比較し、異なれば「他のユーザーが更新しました」という警告を表示する）。
- 4.3のカスケードで自動的に書き換わる後続ノード群については、この基準比較の対象にしない。これらは常にロック取得後の最新値（シートから読み直した値）を起点に計算するため、クライアント側の古いキャッシュに基づく上書きが原理的に発生せず、別途の競合チェックを要しない（4.3参照）。
- 競合が検知された場合、クライアントは競合したノード（および必要であればその直接の親・子）のみをサーバーから再取得して画面を更新する。全件の再読み込みは行わず、他の箇所で進行中の未保存の編集には影響を与えない。
- リアルタイム同期は行わないため、この競合検知は保存操作のタイミングでのみ発生する。
- この楽観的競合検知はNodesシートのみを対象とする。Members・StatusColumns・Dependenciesの編集はLockServiceによる直列化のみで保護し、UpdatedAt照合による警告は行わない（後勝ちで上書きされる）。これらは編集頻度が低く、基本的に1人ずつ操作する運用を前提とするため許容している。実装者はこの非対称性（Nodesのみ競合検知あり）を前提に実装すること。

## 4.7 クライアント側の保存キューと楽観的更新

4.1（要件定義書）の楽観的更新方針を成立させるには、クライアント側の操作順序制御が必要である。google.script.run は非同期APIであり、複数回連続で呼び出した場合にサーバー到着順・応答順がユーザーの操作順と一致する保証はない。この順序のズレを放置すると、後から行った操作の結果が、先に行った操作の（遅れて返ってきた）応答によって上書きされる事故が起こりうる。

- クライアントは、保存系API呼び出し（saveNode・moveNode・deleteNode等）ごとに一意な requestId を採番して送信し、サーバーは応答に同じ requestId を含めて返す。
- Nodesへ直接または波及的に変更を返す保存操作は、対象NodeIdが異なる場合もクライアント全体で1本のチェーンに直列化する。メンバー・ステータス列・依存関係APIのうちNodesを書き換え得る操作も同じチェーンへ参加させる。
- requestIdはキュー投入時に採番する。先行操作の応答を画面へマージした後、応答に含まれたすべてのノードについて後続の未確定patchを投入順に重ね直し、祖先・後続を含む波及応答が新しい楽観的表示を消さないようにする。再適用処理は実行時にstateから最新ノード参照を取り直す。
- 保存中のノードは、UI上で軽い保存中表示（スピナー等）を出し、二重操作を抑止する。
- 保存失敗・競合時のロールバックは、失敗した操作単体（該当ノードの該当フィールド）のみを対象とし、他ノードの表示済み変更には影響を与えない。
- 競合応答は成功コールバックとは別の結果コールバックへ渡す。詳細保存で閉じた入力は競合時に再表示し、削除の競合時は楽観的に除去したサブツリーをスナップショットから戻してからサーバーの最新ノードを重ねる。

## 4.8 Slack通知

- `saveNode` が持つ `statusChanged` 判定にフックしてステータス変更通知を組み立てる。`addComment` はサーバー検証後のMentionsが1件以上ある場合にコメントメンション通知を組み立てる。`saveNode` / `addNode` / `addNodes` は変更前後のAssigneeIds差分から操作本人を除いた新規担当者が1件以上ある場合にアサイン通知を組み立てる。3通知は個別にON/OFFできる。
- `SLACK_NOTIFICATION_SETTINGS.deliveryMode` に `app` または `workflow` を保存する。`app` はIncoming Webhook、`workflow` はWorkflow Builderを使用し、選択方式だけを通知時に参照する。別方式の保存済みURLは、方式別の連携解除を行うまで保持する。
- Incoming Webhook URLは `SLACK_APP_WEBHOOK_URL`、Workflow Builderのステータス用URLは `SLACK_WORKFLOW_STATUS_WEBHOOK_URL`、メンション用URLは `SLACK_WORKFLOW_MENTION_WEBHOOK_URL`、アサイン用URLは `SLACK_WORKFLOW_ASSIGNMENT_WEBHOOK_URL` に保存する。旧 `SLACK_WEBHOOK_URL` は移行互換の読み取りに限定し、更新前から保存されているWorkflowは利用者が項目別変数へ明示的に切り替えるまで `{ text }` 送信を維持する。旧単一Workflow URLは新設のアサイン通知へ流用しない。
- 参照APIはURL本体やトークン断片をブラウザへ返さない。方式・用途ごとの設定済みフラグ、`https://hooks.slack.com/.../••••••` という固定マスク、従来設定フラグを返す。UIは入力欄を空に戻してもこの状態を常時表示し、空欄保存は既存URLを保持する。従来設定フラグがtrueの場合は、現在も通知が継続すること、項目別変数との違い、切り替えに必要なSlack側の作業と用途別手順へのリンクを表示する。
- Slackアプリ方式では、ステータス変更用、メンション用、アサイン用の文面を許可プレースホルダー付き2000文字以内のテンプレートとして保存し、`text` と `blocks` をIncoming Webhookへ送る。未設定時は日本語のデフォルト文面を使い、UIから戻せる。
- Workflow Builder方式では完成済み文面を送らず、ステータス用は `notification_title`, `task_name`, `parent_path`, `before_status`, `after_status`, `actor_name`, `web_app_url`、メンション用は `notification_title`, `task_name`, `parent_path`, `mentioned_user_id`, `mentioned_user_name`, `actor_name`, `comment_text`, `web_app_url`、アサイン用は `notification_title`, `task_name`, `parent_path`, `assigned_user_id`, `assigned_user_name`, `actor_name`, `end_date`, `web_app_url` のフラットなフィールドを送る。`mentioned_user_id` / `assigned_user_id` はSlack側でユーザーID型、ほかはテキスト型とする。
- Workflow Builder方式の設定UIは、上記の送信順に番号を付け、型、用途、値の例を表で示す。加えて「Webhookから」の作成、メッセージ送信ステップ、Slackの「変数を挿入」、Workflow公開、用途別URL保存、接続テストまでを番号付き手順と本文例で案内する。送信処理自体は変数の定義順に依存しないことも明記する。
- Workflow Builder方式のメンション通知は、有効なMembers.SlackUserIdを持つ対象者ごとに1リクエストを送り、Slackのメッセージ送信ステップでユーザーID変数を挿入する。1コメントあたり20人を上限とし、Workflowの1リクエスト/秒制限を避けるため2件目以降を1050ms間隔で送る。
- Workflow Builder方式のアサイン通知も有効なMembers.SlackUserIdを持つ新規担当者ごとに1リクエストを送り、同じ20人上限と1050ms間隔を適用する。Slackアプリ方式では新規担当者を1投稿へまとめる。担当解除、メンバー削除による自動解除、操作本人へのセルフアサインは通知しない。
- `web_app_url` / `{webAppUrl}` には `?node=<NodeId>` を付けたディープリンクを渡す。メンション通知では `&comment=<CommentId>` も付け、`doGet(e)` が検証済みIDをクライアントへ渡す。起動後は対象タスクを開き、コメントIDがあれば対象スレッドを直接取得する。
- Slackアプリ方式のコメント通知で使う `{mentionedUsers}` は、Members.SlackUserIdが有効ならSlack構文 `<@U...>`、未設定ならエスケープ済み表示名を生成する。互換用の `{mentionedNames}` は常に表示名だけを生成する。コメント本文や表示名からSlackの特殊メンションを注入できないよう、`&`・`<`・`>` をエスケープする。
- 旧版で保存されたメンションテンプレートは、初回の設定再保存まで `{mentionedNames}` を `{mentionedUsers}` へ互換移行して、既存案件でも本人通知が有効になるようにする。移行後に利用者が明示的に `{mentionedNames}` を選んだ場合は名前表示として保持する。
- 設定画面では最初に通知方式をカードで選ばせ、選択方式に必要なURL、通知種別ごとのON/OFF、送信文面またはWorkflow変数一覧、用途別接続テスト、方式別連携解除、最終成功・失敗日時、HTTP応答コード、利用者向け失敗理由を提供する。保存・接続テストは対象ボタンと近接した補助表示へスピナーを出す。
- 設定保存・接続テスト・連携解除は同じクライアントキューで直列化し、保存待ちの旧Webhookへ誤ってテスト送信しない。
- LockService のロック中に `UrlFetchApp.fetch` を呼ばない。`saveNode` / `addComment` のロック内では通知payloadの組み立てまで行い、ロック解放後に送信する。
- Slack送信は best-effort とし、Webhook URL が無効、Slack側がエラー、ネットワーク例外のいずれでもタスク保存・コメント保存は成功扱いを維持する。

## 4.9 アプリ内通知ボックス

- `loadAll` はコメント本文を全件返さず、CommentsのCommentId / NodeId / MentionsとNotificationReadsから現在メンバーの未読件数だけを返す。未登録ユーザーは0件とする。
- ベルを開いた時に `getNotifications` で自分宛メンションを新しい順に50件ずつ取得する。カード本文が必要な選択行だけCommentsから復元し、通知一覧は固定サイズ領域内でスクロールする。
- 取得済み通知一覧はクライアントメモリに保持する。30秒以内の再オープンはAPIを呼ばず、TTL経過後も保持中のカードを描画したまま `getNotifications` を実行するstale-while-revalidateとする。`loadAll`で未読数が変化した場合はTTLを無効化する。
- 既読判定は `mention:<CommentId>:<MemberId>` をキーにNotificationReadsへ保存する。単件既読と一括既読はいずれも本人識別後にLockService内で最新Comments.Mentionsを再検証する。
- 通知カードを開く場合は `getNotificationCommentThread` で対象コメントが返信なら親コメントと同じ親配下の返信を、親コメントならその返信群をまとめて返す。クライアントは既存のコメントMapへマージし、対象DOMへスクロールした後に単件既読APIを呼ぶ。
- 未読数はフォーカス復帰・手動同期時の既存 `loadAll` でも更新する。通知専用の定期ポーリングは追加しない。

# 6\. 非機能設計

## 6.1 パフォーマンス・レイテンシ対策

- google.script.run の往復遅延（体感1〜数秒）を前提に、ドラッグ操作やステータス変更はクライアント側で即時反映し、裏側で保存する楽観的更新を基本方針とする（詳細は4.7のクライアント保存キューを参照）。
- AIエージェントによる複数追加は`addNodes`へまとめ、対象解決1回・GAS保存1回とする。既存の`addNode`を並列実行してLockServiceを競合させない。
- 保存失敗・競合時は該当操作のみをロールバックし、エラーを表示する。
- 同時実行数の上限を考慮し、ポーリング等の定期的な自動リクエストは行わない（リアルタイム同期を行わない方針と一致）。
- 最新確認は画面へ戻ったとき（focus / visibilitychange）と利用者がトップバーの同期ピルをクリックしたときに行い、未確定の保存または編集中入力がある場合は全量適用を保留する。
- localStorageキャッシュは案件・ログインユーザーごとに分離し、6時間のTTLを設ける。権限エラー時は削除し、設定画面から手動削除できるようにする。
- 同期応答、保存キューの開始・完了、通知状態更新などでDOMを再描画する場合、同じ表示コンテキストに残る看板列・リスト・モーダル・ドロワー・オーバーレイの縦横スクロール位置を再描画前後で保持する。ビューや設定タブを利用者が明示的に切り替えた場合は、新しいコンテキストを先頭表示してよい。

## 6.2 デザイン方針

- モダンかつ最小限のUIを志向し、説明的な補助文言は避ける。
- 削除確認・競合警告・列削除エラー等のUI文言は、実装フェーズの詳細設計で確定する（要件定義書6章）。

## 6.3 エラーハンドリング方針

- GAS関数呼び出し失敗時は、画面上にエラーメッセージを表示し、必要に応じて再試行またはデータの再読み込みを促す。
- 競合検知・ロック取得失敗時の挙動は4.6の方針に従う。
