# AIエージェント連携 技術仕様

最終更新: 2026-07-22
参照元: `docs/01要件定義・設計/要件定義書.md` / `docs/01要件定義・設計/基本設計書.md` / `AGENTS.md`

## 構成

- 標準transportは、TaskBoard Agent Toolkitが専用Chromeプロファイルで既存Webアプリを開くbrowser transportとする。
- browser transportはHTML Serviceページ内の`google.script.run`を使用し、既存Webアプリと同じ`USER_ACCESSING` / `DOMAIN`の本人セッションでGAS関数を呼ぶ。
- ChromeプロファイルはToolkitの利用者別設定ディレクトリに隔離し、通常利用するブラウザプロファイルを操作しない。
- Apps Script Execution APIと独自OAuthクライアントを使うtransportは、組織管理者が許可・設定できる環境向けの任意オプションとし、TaskBoard標準デプロイでは有効化しない。
- 読み取りには `agentGetContext(payload)` を使用する。
- 書き込みには既存の `addNode(payload)`、`saveNode(payload)`、`addComment(payload)` を使用する。

## `agentGetContext(payload)`

ブラウザ向け `loadAll()` を基に、AIエージェントが対象解決に必要な値だけを返す読み取り専用APIである。派生値、論理削除、作成途中ノードの可視性、本人識別は `loadAll()` と同じ規則に従う。

指定可能なフィルタ:

- `nodeIds`: ノードIDの配列またはCSV。IDによる対象解決では一覧の位置に依存せず、このフィルタを使用する。
- `query`: 名称・説明・成果物・WBS備考の部分一致。大文字小文字を区別しない。
- `statusColumnIds`: ステータス列IDの配列またはCSV。
- `assigneeIds`: 担当メンバーIDの配列またはCSV。
- `leafOnly`: trueの場合は末端ノードだけを返す。
- `limit`: 1〜500。既定値200。

応答には案件ID、現在ユーザー、ノード、メンバー、ステータス列、依存関係、総一致件数、打ち切り有無を含む。Slack設定やコメント本文は含めない。

## 保存時の必須規則

- `saveNode`の前に最新コンテキストを取得し、対象ノードの`updatedAt`を`baseUpdatedAt`として送る。
- 保存系呼び出しごとに一意な`requestId`を送る。
- `CONFLICT`を自動上書きで再試行しない。
- `LOCK_BUSY`だけを、書き込み前に確定した場合に限り最大2回再試行できる。
- タイムアウトは保存済みの可能性があるため自動再試行しない。
- 名前による対象解決が複数件になった場合は失敗させる。

## セキュリティと権限

- Webアプリの`USER_ACCESSING` / `DOMAIN`設定は変更しない。
- browser transportはTaskBoard WebアプリURL以外へ接続せず、専用ChromeのログインCookieを外部へコピーまたは出力しない。
- browser transportの通常操作は画面を表示しないが、初回接続と再認証は利用者が画面上で完了する。
- サービスアカウント、共有APIキー、HMACによる本人偽装、任意コード実行エンドポイントは導入しない。
- 保存系APIの`Session.getActiveUser().getEmail()`と`Members.Email`の照合を省略しない。
