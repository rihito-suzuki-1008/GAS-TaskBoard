## ローカルキャッシュ

- キーはrootIdと現在ユーザーのメールアドレスを含みます。初回読込前は `gasTaskManager.cache.index.v2` に保存したURL・ユーザースコープから案件別キーを引きます。
- `snapshotForCache()` がnodes/members/statusColumns/dependencies/commentCounts/currentMember/rootId/spreadsheetId/cacheScope/cacheUserを保存します。
- キャッシュ適用時は `cacheScope` と `cacheUser` が現在値に一致し、保存から6時間以内のものだけを使います。案件コピー間と共有ブラウザ上のユーザー間でlocalStorageが混在しないようにします。
- 容量超過時は説明・成果物・備考を除いた縮小キャッシュを再保存します。
- setupRequired状態はキャッシュしません。
- 階層の開閉状態はデータキャッシュと分離した表示設定として、WebアプリURL・rootId・現在ユーザーのメールアドレスを含むキーへ保存します。6時間TTLは適用せず、同じブラウザで次回起動時に復元します。「ローカルキャッシュと表示設定を削除」で両方を削除します。

## 自動同期

- 定期ポーリングは行いません。`window.focus` と `visibilitychange` で画面へ戻ったときに同期し、ほかにトップバーの同期ピルから手動の最新確認を提供します。
- 保存中ノードまたはバックグラウンド保存がある場合は `syncStatus = pending` にして同期を中断します。
- リストビューのインライン編集、インラインドロップダウン、インライン日付ピッカー、IME変換、Ganttドラッグ中は全量同期の適用を保留します。入力中DOMを壊さず、次回同期に任せます。
- 同期開始時の `state.mutationGeneration` を保持し、応答適用前に保存中状態・世代差分・Ganttドラッグ中状態・インライン編集中状態を再確認します。汚れていれば全量置換を破棄し、次回同期に任せます。
- 同期ピルのクリックは `syncFromServer({ force: true })` です。通常時は最終同期時刻、保存待ち・ローカル表示・同期中は短い状態名、エラー時は赤いエラー表示へ切り替えます。
- `pendingMutationCount > 0` または `savingNodeIds.size > 0` の間にタブを閉じようとした場合、`beforeunload` で離脱警告を出します。Outbox永続化は実装しません。

## ローカルファースト楽観的更新

- 保存系操作は、ブラウザの `state` を先に更新して `persistLocalCache()` でローカル保存します。
- スプレッドシート保存は `google.script.run` でバックグラウンド実行します。
- 保存完了後は `applyMutationResult()` がサーバー応答をマージし、サーバー側で計算した派生値・リスケジュール結果・最新一覧へ収束させます。
- 保存失敗時は操作ごとのrollbackでローカル状態を戻し、`syncStatus = error` にしてトースト通知します。
- 追加系操作はクライアント側で先にIDを採番します。サーバーは `nodeId` / `clientMemberId` / `clientColumnId` / `commentId` / `dependencyId` を受け取り、同じIDでシートへ保存します。
- クライアント側でも `recomputeClientDerived()` により進捗率、leaf判定、ガント表示期間を再計算します。ただし保存値ではなく表示用派生値です。

## 保存キュー

- `state.nodeMutationChain` はNodesへ直接または波及的に変更を返す操作全体で共有するPromiseチェーンです。異なるNodeIdの保存と、Members / StatusColumns / DependenciesのうちNodesを書き換え得る操作も同じチェーンへ参加します。
- 各ノード操作は発行時にrequestIdを持ちます。
- 先行操作のサーバー応答をマージした後、応答に含まれた全NodeIdに残る後続のoptimistic patchを順に重ね直します。再適用時は `state.nodesById` から最新オブジェクトを取得し、古い参照へ書き込まないようにします。
- optimisticApply後はローカルキャッシュにも保存します。
- `state.backgroundChains` はStatusColumns / Members / Dependencies / Commentsなど、NodeIdを持たない操作をスコープ単位で直列化します。
- `state.pendingMutationCount` はバックグラウンド保存中の件数です。0になるまで自動同期・同期ピルからの最新確認は保留します。
- 未登録ユーザーでもセルフ登録だけは `queueBackgroundMutation('members', 'joinAsMember', ...)` を通します。その他の保存系APIは従来通りcurrentMember必須です。

## 楽観的競合検知

- Nodes系保存だけ `baseUpdatedAt` を照合します。
- `saveNode()` / `moveNode()` / `fitNodeToChildren()` が対象です。
- 詳細ドロワーの保存は、ドロワーを開いた時点の `updatedAt` を `detailDraft.baseUpdatedAt` として保持し、保存時に送ります。
- 競合時はサーバーの最新ノードをstateへマージしつつ、入力中ドラフトは破棄しません。詳細ドロワーが開いていれば `detailDraft.baseUpdatedAt` だけ最新値へ更新し、通常保存で閉じた後に競合した場合は入力内容を復元してドロワーを再表示します。ユーザーが再度保存すると現在の入力内容で上書きできます。
- 削除競合では楽観的に除去したサブツリーをローカルスナップショットから復元してから、競合応答に含まれる最新ノードを上書きします。子孫が次回同期まで消えたままにならないようにします。
- Ganttバーのドラッグ保存は、pointerdown時点の `updatedAt` を送ります。
- Members、StatusColumns、DependenciesはUpdatedAtを持たず、LockServiceによる直列化のみです。

## エラーと再試行

- サーバーの構造化エラーは `[APP:<code>:<retryable>]` 形式でクライアントへ渡します。
- ScriptLockの取得待機上限は15秒です。
- 書き込み前に確定する `LOCK_BUSY` だけを指数バックオフ付きで最大2回試行します。
- タイムアウトは、サーバー側で保存済みなのに応答だけ失われた可能性があるため、自動再試行しません。
- エラー通知は自動消去せず、ユーザーが確認できる状態を保ちます。
