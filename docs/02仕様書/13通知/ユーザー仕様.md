# 通知 ユーザー仕様

最終更新: 2026-07-21
参照元: `docs/01要件定義・設計/要件定義書.md` / `docs/01要件定義・設計/基本設計書.md` / `DESIGN.md` / `AGENTS.md`

## 目的

Slack通知は、タスクのステータス変更、コメントのメンション、新しい担当者へのアサインを、関係者が画面外でも把握できるようにするために使います。通知先は設定したチャンネルであり、対象者へのSlack DMではありません。アプリ内のベルでは、自分宛のコメントメンションを確認できます。

プロジェクト設定では、最初に次のどちらかを選びます。

- Slackアプリ方式：Incoming Webhookを1本使います。装飾と文面をTaskBoard側で管理し、複数人のメンション・アサインも1件の投稿へまとめます。
- Workflow Builder方式：ステータス用、メンション用、アサイン用のWorkflowを最大3本使います。ONにする通知のWorkflowだけを作成し、TaskBoardから項目別の変数を渡します。

## ステータス変更通知

- 看板カードのドラッグ&ドロップでステータスを変更したとき。
- タスク詳細ドロワーでステータスを変更して保存したとき。
- リストビューのインライン操作でステータスを変更したとき。
- 進捗を100%に変更し、保存時に完了ステータスへ移行したとき。

同じステータス列内の並び替えや、子タスクの変更に伴う親タスクの自動ロールアップは通知しません。

## コメントメンション通知

- プロジェクト設定で「コメントのメンションを通知」がONの場合、コメントまたは返信に有効なメンバーのメンションが1件以上含まれると通知します。
- Slackアプリ方式では、複数人を1件の通知へまとめます。Workflow Builder方式では、SlackメンバーID登録済みの対象者ごとに1件ずつ通知します。
- メンション対象MemberIdはコメント保存時にサーバーで再検証します。存在しないメンバーだけが指定されている場合は通知しません。

## アサイン通知

- プロジェクト設定で「新しい担当者へのアサインを通知」がONの場合、タスクの担当者へ新しく追加された人を通知します。
- 担当から外れた人、メンバー削除によって自動的に外れた人、操作した本人へのセルフアサインは通知しません。
- 複数人を同時に追加した場合、Slackアプリ方式では1件へまとめ、Workflow Builder方式ではSlackメンバーID登録済みの対象者ごとに1件送ります。
- タスク名、親パス、新規担当者、操作した人、期限、TaskBoard URLを通知へ利用できます。

## アプリ内通知ボックス

- トップバー右側のベルは、自分宛コメントメンションの通知ボックスです。未読がある場合だけ数字バッジを表示し、0件ではバッジを表示しません。
- ベルを押すと固定サイズの通知ボックスが開き、カード形式の一覧部分だけをスクロールできます。
- 通知カードには投稿者、投稿日時、タスクの階層、コメント本文の抜粋が表示されます。未読カードには未読マークが付きます。
- 未読カードへマウスを重ねるかキーボードフォーカスを置くと、右上にSVGの三点メニューが現れます。「既読にする」でそのカードだけを既読にできます。
- 上部の「一括既読」で、表示ページに限らず現在の自分宛メンションをすべて既読にできます。
- カードを押すと対象タスクの詳細を開き、該当コメントまで自動スクロールします。該当コメントを表示できた後で既読になります。
- 一度読み込んだ一覧はその場でキャッシュされます。30秒以内に閉じて開き直した場合は即座に表示し、それ以降も現在の一覧を表示したまま裏で最新化します。
- 既読状態はスプレッドシートへメンバー別に保存されるため、別のブラウザや端末でも共有されます。機能追加前の過去メンションも一覧に含まれます。

## 通知内容と送信文面

- Slackアプリ方式には、ステータス変更用、コメントメンション用、アサイン用の個別のデフォルト文面があります。
- Slackアプリ方式ではプロジェクト設定の「Slack通知」で文面を編集し、「デフォルトに戻す」で初期文面へ戻せます。Workflow Builder方式ではSlackの「チャンネルにメッセージを送信」ステップで本文と装飾を編集します。
- ステータス変更文面では `{taskName}`, `{parentPath}`, `{beforeStatus}`, `{afterStatus}`, `{actorName}`, `{webAppUrl}` を使えます。
- メンション文面では `{taskName}`, `{parentPath}`, `{mentionedUsers}`, `{mentionedNames}`, `{actorName}`, `{commentText}`, `{webAppUrl}` を使えます。`{mentionedUsers}` はSlackメンバーID登録済みの対象者を本人メンションにします。
- アサイン文面では `{taskName}`, `{parentPath}`, `{assignedUsers}`, `{assignedNames}`, `{actorName}`, `{endDate}`, `{webAppUrl}` を使えます。`{assignedUsers}` はSlackメンバーID登録済みの新規担当者を本人メンションにします。
- Slack通知の `{webAppUrl}` / `web_app_url` は対象タスクを直接開くURLです。コメントメンション通知では該当コメントまで自動スクロールします。
- `{commentText}` はSlack通知用に先頭1200文字まで差し込まれます。コメント本体の保存内容は省略されません。
- 文面は2000文字以内です。未対応のプレースホルダーを含む場合は保存できません。

## 通知されない操作

- タスク名、優先度、説明、日付だけの変更。アサイン通知がOFFの場合の担当者変更。
- カスケードリスケジュールによる日程変更。
- アサインを伴わないタスク追加・削除。
- メンションを含まないコメント・返信。
- 「コメントのメンションを通知」がOFFの場合のコメント・返信・メンション。

## 未設定時

- 選択中の方式で、ONにした通知に必要なWebhook URLが未設定の場合、その設定は保存できません。
- 未設定でもアプリの保存操作や画面表示は変わりません。

## 設定と確認

- プロジェクト設定は左側の縦タブから「Slack通知」を選び、右側で通知方式と必要なWebhook URLを設定します。
- ステータス変更通知、コメントメンション通知、アサイン通知のON/OFF、方式・用途ごとの接続テスト、連携解除を同じ画面で操作できます。
- Webhook URL本体は保存後にブラウザへ返しません。ただし各URL欄には「設定済み／未設定」と伏せ字の保存状態が常に表示されます。入力欄が空でも設定済み表示があれば保存されています。空欄のまま再保存してもURLは消えません。
- 保存中と接続テスト中は、操作したボタンと近くの状態欄にスピナーが表示されます。
- Workflow Builder方式では、ONにする通知に応じてステータス変更用、コメントメンション用、アサイン用を作成し、発行されたURLを同じ番号の入力欄へ保存します。
- 画面内に、各Workflowの作成手順、変数の追加順、型、値の例、メッセージ本文例を表示します。変数の送信処理は並び順に依存しませんが、画面どおりの順に追加すると設定を照合しやすくなります。
- 最終送信成功日時、最終失敗日時、HTTP応答コード、失敗理由を設定画面で確認できます。
- Slack送信が失敗してもタスク保存やコメント保存は取り消されません。

### Workflow Builder方式の共通手順

1. Slackで「その他」→「ツール」→「ワークフロー」を開き、「新規」→「ワークフローを構築」を選びます。
2. 開始方法で「Webhookから」を選びます。表示されない場合は、Slackのプランまたはワークスペース権限を確認します。
3. 画面の表にある変数を上から順に追加します。変数名は大文字・小文字とアンダースコアを含めて完全一致させます。
4. 「チャンネルにメッセージを送信」ステップを追加し、投稿先を選びます。
5. 本文例の変数は文字として入力せず、Slackの「変数を挿入」から同名の変数を選びます。太字・箇条書き・リンクなどはSlackの本文編集画面で付けます。
6. Workflowを公開し、発行された `/triggers/...` URLを対応する入力欄へ保存してから接続テストを実行します。

### ステータス変更Workflowの変数

| 順番 | 変数名 | 型 | 内容 |
|---:|---|---|---|
| 1 | `notification_title` | テキスト | 通知タイトル |
| 2 | `task_name` | テキスト | タスク名 |
| 3 | `parent_path` | テキスト | 親タスクの階層 |
| 4 | `before_status` | テキスト | 変更前ステータス |
| 5 | `after_status` | テキスト | 変更後ステータス |
| 6 | `actor_name` | テキスト | 操作した人 |
| 7 | `web_app_url` | テキスト | TaskBoardを開くURL |

本文では、通知タイトル、タスク名、親パス、変更前後のステータス、実行者、TaskBoardのURLを必要な順に「変数を挿入」から配置します。

### コメントメンションWorkflowの変数

| 順番 | 変数名 | 型 | 内容 |
|---:|---|---|---|
| 1 | `notification_title` | テキスト | 通知タイトル |
| 2 | `task_name` | テキスト | タスク名 |
| 3 | `parent_path` | テキスト | 親タスクの階層 |
| 4 | `mentioned_user_id` | ユーザーID | 通知するSlack本人 |
| 5 | `mentioned_user_name` | テキスト | メンション対象者名 |
| 6 | `actor_name` | テキスト | コメント投稿者 |
| 7 | `comment_text` | テキスト | コメント本文 |
| 8 | `web_app_url` | テキスト | TaskBoardを開くURL |

`mentioned_user_id` だけを「ユーザーID」型にし、それ以外は「テキスト」型にします。本文へ `mentioned_user_id` をSlackの変数として挿入すると、Membersに登録したSlackメンバーIDの本人へメンション通知されます。

### アサイン通知Workflowの変数

| 順番 | 変数名 | 型 | 内容 |
|---:|---|---|---|
| 1 | `notification_title` | テキスト | 通知タイトル |
| 2 | `task_name` | テキスト | タスク名 |
| 3 | `parent_path` | テキスト | 親タスクの階層 |
| 4 | `assigned_user_id` | ユーザーID | 新しく担当へ追加されたSlack本人 |
| 5 | `assigned_user_name` | テキスト | 新規担当者名 |
| 6 | `actor_name` | テキスト | 操作した人 |
| 7 | `end_date` | テキスト | タスクの期限。未設定時は「未設定」 |
| 8 | `web_app_url` | テキスト | TaskBoardを開くURL |

`assigned_user_id` だけを「ユーザーID」型にし、それ以外は「テキスト」型にします。本文へ `assigned_user_id` をSlackの変数として挿入すると、新規担当者本人へメンション通知されます。

### TaskBoard更新前のWorkflow設定

更新前から保存されているWorkflow URLは、完成した通知文を `text` という1項目にまとめる従来設定で送信を続けます。設定画面では「従来設定で動作中」と表示し、通知が停止していないこと、現在の項目別変数へ切り替えるために必要なSlack側の作業、対応する手順へのリンクを併記します。切り替える場合は、上記の変数を登録したWorkflowを作成または更新し、Slackが発行したURLを対応する欄へ入力して保存します。
