## 送信条件

- `saveNode()` で `StatusColumnId` が変更された場合だけステータス変更通知の対象にします。
- 看板のカードD&D、詳細ドロワー、リストのインライン編集、進捗100%による完了移行は、いずれも `saveNode()` の同じ判定を通ります。
- `addComment()` でサーバー検証後のMentionsが1件以上あり、`mentionEnabled` がtrueの場合はメンション通知を送ります。
- `saveNode()` / `addNode()` で変更前後のAssigneeIds差分に新規担当者があり、`assignmentEnabled` がtrueの場合はアサイン通知を送ります。操作本人は差分から除外します。
- アサインを伴わないタスク追加、担当解除、日付変更、説明変更、カスケードリスケジュール、メンションなしのコメントは通知対象外です。

## Webhook設定

- `SLACK_NOTIFICATION_SETTINGS.deliveryMode` は `app` / `workflow` のいずれかです。
- URLはScript Propertiesへ分離保存します。Slackアプリ方式は `SLACK_APP_WEBHOOK_URL`、Workflow Builder方式は `SLACK_WORKFLOW_STATUS_WEBHOOK_URL`、`SLACK_WORKFLOW_MENTION_WEBHOOK_URL`、`SLACK_WORKFLOW_ASSIGNMENT_WEBHOOK_URL` です。旧 `SLACK_WEBHOOK_URL` は移行互換用で、アサイン用途へは流用しません。
- 設定参照APIはWebhook URL本体を返さず、方式・用途ごとの設定済みフラグ、固定マスク表示、旧形式フラグ、通知ON/OFF、テンプレート、配信状態だけを返します。
- 通知ON/OFFとテンプレートは `SLACK_NOTIFICATION_SETTINGS`、最終配信状態は `SLACK_DELIVERY_STATUS` にJSON保存します。
- `SLACK_NOTIFICATION_SETTINGS` は `deliveryMode`, `statusChangeEnabled`, `mentionEnabled`, `assignmentEnabled`, `statusTemplate`, `mentionTemplate`, `assignmentTemplate` を持ちます。既存データに新しいキーがない場合は旧URLの種類から方式を推定し、アサイン通知はOFF、ほかはデフォルト値を補います。
- 未設定の場合は送信処理全体をスキップします。
- 設定変更・接続テスト・連携解除はMembersに登録されたユーザーだけが実行できます。
- 設定変更・接続テスト・連携解除はクライアントの `slack-settings` キューで直列化します。Webhook保存中に接続テストを押しても、保存完了後のWebhookへ送信します。

## 送信テンプレート

- 次のテンプレート規則はSlackアプリ方式に適用します。テンプレートは2000文字以内、Slackへ渡す展開後メッセージは3000文字以内です。
- ステータス変更用の許可キーは `taskName`, `parentPath`, `beforeStatus`, `afterStatus`, `actorName`, `webAppUrl` です。
- メンション用の許可キーは `taskName`, `parentPath`, `mentionedUsers`, `mentionedNames`, `actorName`, `commentText`, `webAppUrl` です。
- アサイン用の許可キーは `taskName`, `parentPath`, `assignedUsers`, `assignedNames`, `actorName`, `endDate`, `webAppUrl` です。
- `commentText` は通知payloadでは先頭1200文字に制限し、長いコメントでも後続のWebアプリURLなどがSlackのsection上限で欠落しにくいようにします。
- 未対応キーは設定保存時に拒否し、保存済みJSONが壊れている場合はデフォルト文面へフォールバックします。
- 差し込み値はSlack mrkdwn用にエスケープし、コメント本文由来の `<!channel>` などが意図せずSlack特殊メンションにならないようにします。

## ロックと失敗時の扱い

- `saveNode()` と `addComment()` はLockServiceのロック内で保存と通知payloadの組み立てまで行います。
- `UrlFetchApp.fetch()` はロック解放後に実行します。
- `postToSlack_()` は `muteHttpExceptions: true` を指定し、HTTPステータスと本文を検査します。
- Slack側の失敗、Webhook URL不正、ネットワーク例外のいずれも `saveNode()` / `addComment()` の成否には影響させません。
- 成否と応答コードをScript Propertiesへ記録し、404/403/429や代表的なSlackエラー本文を利用者向けメッセージへ変換します。
- 通知を試行したAPI応答には最新の公開Slack設定を含め、クライアントの配信状態を更新します。

## payload

- Slackアプリ方式はplain textとBlock Kitのsectionを同じ展開済みテンプレートから組み立て、Incoming Webhookへ1リクエスト送ります。
- Workflow Builder方式のステータスpayloadは `notification_title`, `task_name`, `parent_path`, `before_status`, `after_status`, `actor_name`, `web_app_url` のフラットな文字列フィールドです。
- Workflow Builder方式のメンションpayloadは `notification_title`, `task_name`, `parent_path`, `mentioned_user_id`, `mentioned_user_name`, `actor_name`, `comment_text`, `web_app_url` のフラットなフィールドです。`mentioned_user_id` はSlackでユーザーID型として定義します。
- Workflow Builder方式のアサインpayloadは `notification_title`, `task_name`, `parent_path`, `assigned_user_id`, `assigned_user_name`, `actor_name`, `end_date`, `web_app_url` のフラットなフィールドです。`assigned_user_id` はSlackでユーザーID型として定義します。
- メンションWorkflowとアサインWorkflowは有効なSlackUserIdを持つ対象者ごとに1リクエストを作り、最大20件、2件目以降は1050ms間隔で送ります。
- 更新前から保存されているWorkflow URLは、利用者が項目別変数へ明示的に切り替えるまで `{ text }` payloadで通知を継続します。公開設定には従来設定フラグを付け、UIで現在の動作、項目別変数との違い、切り替え手順を説明します。

## 設定UI

- プロジェクト設定モーダルは最大幅1120pxで、左の縦タブと右の内容領域に分けます。
- 通知方式を先にカードで選び、選択方式に関係する項目だけを表示します。
- URL入力欄とは別に設定済み状態と固定マスクを表示します。URL入力は置き換え専用で、空欄は既存値維持です。
- `dialog.submitting` と `dialog.slackAction`（`save`, `test:app`, `test:workflowStatus`, `test:workflowMention`, `test:workflowAssignment`）を使い、対象操作へローカルなスピナーを表示します。
- アサイン接続テストでは `dialog.slackAction = 'test:workflowAssignment'` を使用し、実行者本人のSlackUserIdをテスト宛先にします。

## アプリ内通知と既読状態

- `NotificationReads` 列は `NotificationKey`, `RecipientMemberId`, `NotificationType`, `SourceId`, `ReadAt` です。全列をテキスト書式に固定します。
- 通知イベント自体は複製保存しません。Comments.Mentionsに現在メンバーを含む有効ノードのコメントを通知とし、`mention:<CommentId>:<MemberId>` の既読行がなければ未読です。
- `loadAll()` は通知本文を含めず、現在メンバーの `notificationUnreadCount` だけを返します。
- `getNotifications()` は軽量なコメント列から自分宛メンションを抽出し、新しい順に50件ずつ選んだ後、選択行の本文だけを復元します。
- 通知一覧はクライアントメモリへ30秒間キャッシュします。TTL経過後は一覧を消さず `notificationRefreshing` 状態で再取得し、`loadAll()` の未読数が変化したときはキャッシュを期限切れ扱いにします。
- `markNotificationRead()` は通知キー、コメントID、現在メンバーの一致とComments.Mentionsをロック内で検証し、既存行がなければ既読行を追加します。
- `markAllNotificationsRead()` は有効ノード上の未読メンションをすべて列挙し、既読行を一括追加します。表示中ページだけに限定しません。
- `getNotificationCommentThread()` は自分宛メンションを検証し、対象が返信なら親コメントと同じ親配下の返信、対象が親ならその返信群を返します。
- クライアントは対象スレッドを既存コメントMapへマージし、`data-comment-id` の要素へスクロールした後で単件既読を呼びます。通常ページ取得と直接取得が競合しても対象スレッドを失わないよう、どちらも既存Mapへマージします。
- Slack用リンクは `webAppUrlForNode_()` が `?node=<NodeId>` と、メンション時の `&comment=<CommentId>` を付与します。`doGet(e)` はID文字種と長さを検証してHTMLブートストラップへ渡し、クライアントは `getCommentThread()` で古いコメントも対象スレッドを直接取得します。
