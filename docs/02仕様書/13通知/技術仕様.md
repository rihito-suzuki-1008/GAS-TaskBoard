## 送信条件

- `saveNode()` で `StatusColumnId` が変更された場合だけ通知対象にします。
- タスク追加、日付変更、説明変更、カスケードリスケジュール、コメント、メンションは通知対象外です。
- 看板のカードD&Dと詳細ドロワーのステータス変更は、どちらも `saveNode()` を通るため同じ通知経路です。

## Webhook設定

- Webhook URL はプロジェクト設定UIから受け取り、Script Properties の `SLACK_WEBHOOK_URL` に保存します。
- 設定参照APIはWebhook URL本体を返さず、マスク表示、通知ON/OFF、配信状態だけを返します。
- 通知ON/OFFは `SLACK_NOTIFICATION_SETTINGS`、最終配信状態は `SLACK_DELIVERY_STATUS` にJSON保存します。
- 未設定の場合は送信処理全体をスキップします。
- 設定変更・接続テスト・連携解除はMembersに登録されたユーザーだけが実行できます。
- 設定変更・接続テスト・連携解除はクライアントの `slack-settings` キューで直列化します。Webhook保存中に接続テストを押しても、保存完了後のWebhookへ送信します。

## ロックと失敗時の扱い

- LockService のロック内では通知payloadの組み立てまで行います。
- `UrlFetchApp.fetch()` はロック解放後に実行します。
- `postToSlack_()` は `muteHttpExceptions: true` を指定し、HTTPステータスと本文を検査します。
- Slack側の失敗、Webhook URL不正、ネットワーク例外のいずれも `saveNode()` の成否には影響させません。
- 成否と応答コードをScript Propertiesへ記録し、404/403/429や代表的なSlackエラー本文を利用者向けメッセージへ変換します。

## payload

- plain text と Block Kit の両方を含めます。
- 内容はタスク名、親パス、変更前後の列名、実行者名、WebアプリURLです。
