# AGENTS.md

このリポジトリは Google Apps Script（GAS）で構築する社内向けタスク管理アプリです。
コーディングを始める前に、必ず `docs/01要件定義・設計/要件定義書.md` と `docs/01要件定義・設計/基本設計書.md` を読んでください。
この AGENTS.md は、その2文書を実装に落とし込む際に **絶対に踏み外してはいけない一線** をまとめたものです。
2文書は3回の社内レビューを経ており、ここに書かれているルールの多くは実際に起きた設計事故（起動デッドロック・競合による更新のロストなど）への対策です。事情を知らずに「シンプルにしよう」と省略すると、レビューで一度は塞いだ穴が再発します。

---

## 1. プロジェクト概要

- Google スプレッドシートにコンテナバインドされた GAS プロジェクト。1スプレッドシート＝1案件。
- エンドユーザーUIは `doGet(e)` で配信する独立した Web アプリ（HTML Service）。スプレッドシートのメニュー／サイドバーは使わない。
- データストアはスプレッドシートの各シート（Nodes / Members / StatusColumns / Dependencies / Comments / ActivityLog / Milestones / Meetings / CalendarOverrides / NotificationReads）。
- 看板・ガント（フル機能）・リストの3ビュー＋タスク詳細モーダル＋メンバー管理画面。

詳細な機能仕様は `docs/01要件定義・設計/要件定義書.md`、データモデル・画面設計・ロジック設計は `docs/01要件定義・設計/基本設計書.md` を参照。本ファイルとそれらの記述が矛盾する場合は、2文書を正としてこのファイルを更新すること。

## 2. 技術スタック・コマンド

- サーバーサイド: Google Apps Script（V8ランタイム）
- フロントエンド: **Vanilla JavaScript が基本方針**。理由は保守性・依存管理の簡素化・デプロイ容易性であり、CSP制約が理由ではない（重量フレームワークを提案しない）。
- 開発は [clasp](https://github.com/google/clasp) を前提とする。
  - `clasp push` : ローカルの変更をGASプロジェクトへ反映
  - `clasp open` : Apps Scriptエディタを開く
  - `clasp deploy` : Webアプリとしてデプロイ（`/exec` URLの更新）
- テストは可能な範囲で `clasp run` によるサーバー関数の単体呼び出し、またはローカルでロジックをNode.js関数として切り出してテストすること（GAS特有API部分はモック化）。

## 3. 絶対に守るルール（アーキテクチャ原則）

これらは全て過去のレビューで実際に検出された不具合への対策です。省略・簡略化しないこと。

1. **派生値は保存しない。常に動的計算する。**
   親ノードの進捗率とガントの拡張表示期間は Nodes シートに保存しない。読み込み・保存のたびにサーバーサイドで再計算する。これにより「複数人が別々の子ノードを編集して親の行を競合させる」事故を構造的に防いでいる（基本設計書2.9）。末端ノードの手動進捗 `Progress` はユーザー入力値として保存してよいが、親ノードの進捗・ロールアップ期間をキャッシュ保存する変更は、キャッシュ無効化のタイミングまで含めて設計し直さない限り行わないこと。

2. **カスケード処理は、書き込み直前にシートから最新値を読み直す。クライアントのキャッシュは絶対に信用しない。**
   依存関係の自動連鎖リスケジュール（基本設計書4.3）は、LockService のロック取得後にシートの現在値を再読み込みしてから計算する。これを怠ると「Aさんの古い画面がBさんの確定済み変更を上書きする」ロストアップデートが再発する。

3. **リスケジュールはトポロジカル順＋DAG対応。一本道の依存関係だけでテストして満足しないこと。**
   後続タスクが複数の先行タスクを持つ合流構造で、「全先行タスクの終了日の最大値」を基準に判定する（基本設計書4.3）。テストケースには必ず合流パターン（P1・P2 → S）を含めること。

4. **「常に隣接」ではなく「崩れたときだけ、崩れた分だけ」繰り下げる（制約型）。**
   ユーザーが意図的に空けたバッファ（余白）を、リスケジュールで勝手に潰さないこと。

5. **カスケードで書き換えた行は UpdatedAt / UpdatedBy も必ず更新する。**
   これを怠ると、後から別ユーザーが保存する際の楽観的競合検知が発火しない。

6. **本人識別はメールアドレス経由。`Session.getActiveUser().getEmail()` と `Members.Email` の照合で行う。**
   本人識別チェックの例外は、初回セットアップ処理（基本設計書2.11）とセルフ登録（`joinAsMember`。登録できるのはセッションのメール本人のみ）の2つだけ。それ以外の全保存系関数は本人識別必須。

7. **初回セットアップはルートノード・初期ステータス列（完了フラグ付き）・初期メンバー登録を一括で行う。**
   これが無いと、スプレッドシートをコピーしただけの新品案件が起動不能になる（実際に起きた不具合）。

8. **Members の最後の1件は削除できないようにする。**
   本人識別の基盤が消えるとアプリ全体が保存不能になるため。

9. **日付は `YYYY-MM-DD` の文字列で統一。ネイティブDate型を使わない。**
   スプレッドシートは書式指定なしだとテキストを日付値に自動変換するため、Nodes/Comments/Milestones/ActivityLog/CalendarOverrides/NotificationReads の日付・日時列は `setNumberFormat("@")` でテキスト書式に固定すること。WBSシートは一方向の出力アーティファクトであり、日数式を成立させるため日付値＋表示書式で出力してよい。

10. **開始日・終了日は任意項目。未設定ノードはロールアップ計算から除外し、依存関係の先行/後続には設定不可。**

11. **依存関係は末端ノード（子を持たないノード）同士のみ。** 自己参照・重複・循環は禁止。追加時にも即座に整合チェック・リスケジュールを実行する。

12. **ノード削除はソフトデリート。** `DeletedAt`/`DeletedBy` を立てるだけで、行やComments/Dependenciesの関連行は物理的に消さない。全ての読み込み系APIは `DeletedAt` が空でないノードを除外する。

13. **保存系APIの戻り値は「自分だけ」ではなく「影響が及ぶ全ノード」を返す。**
    ステータス変更→祖先の進捗が変わる、日程変更→後続チェーンの日程が変わる、といった波及を、クライアントが追加のリクエストなしで画面に反映できるようにする（基本設計書5章のAPI契約を参照）。

14. **クライアント側に保存キューを実装する（基本設計書4.7）。**
    `google.script.run` は非同期・順序保証なし。同一ノードへの連続保存は直列化し、`requestId` を使って古い応答を破棄すること。これを省略すると「後から行った操作が、遅れて返ってきた古い保存結果で上書きされる」事故が起きる。

15. **楽観的競合検知は Nodes シートのみ。** Members / StatusColumns / Dependencies / ActivityLog / Milestones / Meetings / CalendarOverrides / NotificationReads は LockService による直列化のみで、UpdatedAt照合は行わない（意図的な非対称性。全シートに同じ仕組みを足そうとしないこと）。

16. **完了列（IsDoneColumn）は常に厳密に1件のみ true。** 削除・切り替え操作で0件または複数件trueになる状態を一瞬たりとも許可しないこと。

17. **アクセス権限・デプロイ設定は変更しない。**
   実行モードは「アクセスしたユーザーとして実行」、対象は同一Google Workspaceドメイン全員、スプレッドシートは全員に編集権限を付与——これは直接シート編集による迂回リスクを承知の上での意識的な選択（基本設計書1.2）。勝手に「オーナー実行」へ変更したり、シートを保護したりしないこと。変更したい場合は必ず先に相談する。

18. **WBSシートは一方向の出力アーティファクト。**
    アプリは `WBS` シートを読まない。`HEADERS` / `ensureSchema_` / `loadAll` に含めず、正本は常にアプリの各データシートとする。`exportWbs` は長時間の書式出力で全ユーザーの保存を止めないよう、意図的に `withLock_` を使わない。

19. **案件カレンダーはガントの表示専用。**
    CalendarOverrides には手動の `working` / `holiday` だけを保存し、「標準」は行を削除して自動判定へ戻す。タスク期間・日数、依存関係の制約判定や自動リスケジュール、親期間のロールアップ、WBS出力へは使用しない。

## 4. データモデル早見表

| シート | 役割 |
|---|---|
| Nodes | プロジェクト・グループ・タスクを ParentId 参照で統一管理。Type列は持たない（子を持つかどうかは実行時判定） |
| Members | 担当者・投稿者候補。Email列で本人と照合し、任意のSlackUserIdでSlack本人メンションと紐づける |
| StatusColumns | 看板の列定義。IsDoneColumnは常に1件のみtrue |
| Dependencies | 末端ノード間の先行・後続関係 |
| Comments | ノードごとのコメント（AuthorNameを投稿時点でスナップショット保存） |
| ActivityLog | ステータス・進捗変更の追記専用ログ |
| Milestones | WBS出力用マイルストーン |
| Meetings | WBS出力用会議体 |
| CalendarOverrides | ガント表示用の案件共通カレンダー例外（Dateごとにworking / holiday） |
| NotificationReads | Comments.Mentionsから動的に構成する通知のメンバー別既読記録。未読通知そのものは保存しない |

詳細な列定義は `docs/01要件定義・設計/基本設計書.md` 2章を参照。

## 5. 推奨実装順序

看板・リスト・詳細モーダル・メンバー管理を先に完成させてから、最後にガントへ着手すること。ガントは本アプリで最も実装リスクが高い機能（基本設計書3.3の注記）。

1. Nodes/Members/StatusColumns の基本CRUD、初回セットアップ、本人識別
2. 看板ビュー（列カスタマイズ含む）
3. リストビュー
4. タスク詳細モーダル（コメント含む）
5. メンバー管理画面
6. クライアント保存キューの実装（4章のP0ルール14）
7. ガント：①バー表示のみ → ②ドラッグ移動・リサイズ → ③依存関係の矢印描画 → ④自動連鎖リスケジュール（この順で段階的に検証する）

## 6. 仕様に迷ったら

`docs/01要件定義・設計/要件定義書.md` 6章「オープン事項」に、意図的に未決定のまま残した項目（文字数上限、UI文言、依存関係追加のUI詳細など）がある。これらは実装者の裁量で妥当な値を決めてよいが、決めた内容はコードコメントかREADMEに残すこと。

上記の「絶対に守るルール」に関わる部分（データ整合性・競合検知・削除方式など）は、たとえ実装が面倒でも省略や自己判断での簡略化をしないこと。判断に迷う場合は、実装を進める前に人間に確認すること。
